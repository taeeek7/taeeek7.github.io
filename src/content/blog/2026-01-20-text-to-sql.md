운영 담당자가 데이터 하나 보려고 매번 개발팀에 SQL을 요청한다. "이번 주 달빛하우스 지점 청소 몇 건 끝났어요?" 같은 질문이 슬랙으로 온다. 요청하면 빨라야 몇 시간, 늦으면 하루. 그 사이에 의사결정은 멈춰 있다. 개발팀 입장에서도 이건 반복되는 잡일이고, 물어보는 쪽은 눈치가 보인다. 이 구조가 양쪽 다 손해라는 게 계속 걸렸다.

이걸 몇 번 지켜보다가, "그냥 자연어로 물어보면 SQL로 바꿔서 답해주는 걸 만들면 되지 않나?" 싶어서 시작했다. 결론부터 말하면 프로토타입까지는 잘 굴러갔고, 프로덕션에 올리기 전에 멈췄다. 근데 그 과정에서 RAG랑 프롬프트 엔지니어링을 처음 제대로 만져봤고, 무엇보다 "어디서 멈춰야 하는지"를 배워서 정리해둔다.

## 뭘 만들고 싶었나

목표는 단순했다. 이런 걸 하고 싶었다.

> "달빛하우스 지점의 오늘 대기중인 업무 조회해줘"

이 문장을 받아서 → SQL로 바꾸고 → DB에서 실행하고 → 결과를 다시 자연어로 요약해서 답해주는 것. 파이프라인 자체는 뻔하다. LangChain으로 엮고, LLM은 Gemini를 붙였다.

문제는 이 뻔한 파이프라인의 중간, "자연어 → SQL" 한 칸에 다 몰려 있었다. 그리고 그 칸을 뜯어보니 문제가 두 가지로 갈렸다.

1. **스키마가 크다.** 우리 DB는 운영에 필요한 것만 추려도 테이블이 20개쯤 되고, 엔티티 관계가 꽤 복잡하다. ticket 하나 조회하려 해도 room, room_group, keeper를 경유해야 한다. 이 스키마를 통째로 LLM한테 넘기면 토큰도 아깝고, 정작 정확도는 떨어진다. 관계없는 테이블 정보가 노이즈가 돼서 오히려 이상한 JOIN을 유도한다.
2. **사람 말과 DB 값이 다르다.** "달빛하우스"는 사람이 부르는 이름이고, DB 안에서는 `RGRP0…` 형태의 내부 ID다. "대기중"은 `PENDING`이고, "완료"는 `RESOLVED`다. 이 변환을 LLM이 알 리가 없다. 스키마만 줘서는 절대 풀 수 없는 문제였다.

이 두 개를 각각 어떻게 풀지가 이 프로젝트의 설계 전부였다고 봐도 된다.

## 스키마를 다 넣지 말고, 관련된 것만 넣자 (RAG)

처음엔 그냥 스키마를 다 프롬프트에 때려넣었다. 20개 테이블의 컬럼 정의를 전부. 역시나 별로였다. 토큰은 토큰대로 먹고, "업무 조회" 질문에 엉뚱하게 keeper_level_group 같은 걸 끌어와서 조인을 걸었다. LLM은 눈앞에 있는 정보를 다 쓰려고 하지, 필요한 것만 고르지 않는다.

그래서 RAG를 붙였다. 발상은 단순하다. **질문마다 필요한 테이블은 몇 개 안 된다. 그러면 그 몇 개만 찾아서 넣자.**

- 20개 테이블 스키마를 벡터 DB(ChromaDB)에 임베딩해서 넣어두고
- 질문이 들어오면 관련도가 높은 **상위 5개 테이블만** 검색해서 프롬프트에 넣는다

한국어 질문을 다뤄야 하니까 임베딩 모델은 다국어 지원되는 걸(`paraphrase-multilingual-MiniLM`) 골랐다. 영어 전용 모델로는 "키퍼 등급"과 keeper_level 테이블 설명이 가깝게 붙질 않았다. 이렇게 하니까 "업무 조회" 질문엔 ticket 관련 테이블이, "키퍼 등급" 질문엔 keeper 관련 테이블이 딸려 나왔다. 스키마를 다 넣던 것보다 확실히 정확해졌고, 프롬프트도 훨씬 가벼워졌다.

여기서 중요한 설계 판단이 하나 있었다. **벡터 DB에 넣는 건 테이블 스키마만이 아니다.** 스키마, 엔티티 매핑, 코드 매핑을 전부 같은 벡터 스토어에 넣고, 질문 하나로 세 종류를 동시에 검색하게 했다. 질문에 "달빛하우스"가 있으면 지점 매핑이 딸려 나오고, "대기중"이 있으면 상태 코드 매핑이 딸려 나온다. 검색 한 방에 "어떤 테이블 + 어떤 값"이 같이 붙어 나오는 구조다. 이게 두 번째 문제로 이어진다.

## 자연어 → DB 값 매핑을 어떻게 저장할까

이게 은근히 노가다였고, 은근히 설계가 필요했다. "달빛하우스"가 `RGRP0…`로 시작하는 내부 ID라는 걸 LLM이 알 리가 없으니, 이 대응표를 어딘가에 만들어서 검색되게 해야 한다. 근데 매핑에도 종류가 있었다. 나는 이걸 두 가지로 쪼갰다.

**엔티티 매핑** — 지점, 키퍼처럼 실제 레코드의 이름↔ID 대응이다. 이건 DB에서 자동으로 긁어올 수 있다.

```json
{
  "type": "entity_mapping",
  "entity": "room_group",
  "mappings": [
    {
      "id": "RGRP0XXXXXXXX…",
      "name": "달빛하우스",
      "aliases": ["달빛", "달빛하우스"]
    }
  ]
}
```

**코드 매핑** — 상태값, 업무 유형 같은 ENUM 대응이다. 이것도 코드 자체는 DB에서 긁어오지만, "대기중"이라는 한국어 별칭은 사람이 알려줘야 한다.

```
"대기중"  → PENDING
"완료"    → RESOLVED
"일반"    → TICKET_TYPE1
"검수"    → TICKET_TYPE4
```

여기서 핵심이 `aliases`(별칭)였다. 사람은 "달빛하우스"를 "달빛"이라고도 부르고 "완료"를 "끝난 거"라고도 한다. 이름 하나만 넣어두면 검색이 안 걸린다. 그래서 매핑마다 별칭 배열을 붙였고, 이걸 벡터 검색이 물어올 수 있게 했다. 이렇게 만든 매핑이 250개쯤 됐다. 질문에서 이런 자연어가 나오면 관련 매핑이 검색되고, 프롬프트가 그걸 보고 DB 값으로 바꿔서 쿼리를 짠다.

## 이 매핑을 매번 손으로 관리할 순 없다 (동기화 배치)

문제는 이거였다. 지점은 계속 새로 생기고, 테이블에 컬럼도 붙는다. 250개 매핑과 20개 스키마를 손으로 관리하는 건 며칠 못 간다. 스키마 하나 바뀔 때마다 벡터 DB를 사람이 다시 빌드해야 한다면, 이건 만들자마자 죽은 시스템이다.

그래서 메타데이터를 통째로 자동 동기화하는 배치를 설계했다. 이게 이 프로젝트에서 제일 공들인 부분이다. 흐름은 이렇다.

1. **스키마 동기화** — `information_schema`를 조회해서 현재 DB의 테이블·컬럼을 긁어온다. 기존 메타데이터와 비교해서 신규는 INSERT, 사라진 컬럼은 지우지 않고 `is_active = 0`으로 soft delete. 이력을 남기려고 그랬다.
2. **엔티티 매핑 동기화** — room_group, keeper 같은 대상 테이블에서 id·name·status를 긁어온다. 신규 지점이 생기면 자동으로 매핑에 추가되고, 별칭은 일단 이름 그대로 하나 넣어둔다.
3. **메타데이터 생성(LLM)** — 여기가 재밌다. 테이블 설명, 컬럼 설명, 별칭은 사람이 다 못 쓴다. 그래서 description이 비어 있는 항목만 골라서 LLM한테 "이 테이블/컬럼 이름 보고 한글 설명이랑 별칭 만들어줘"라고 시켰다. 컬럼명과 타입만 줘도 웬만한 설명은 나온다. 다만 코드값 설명(`TICKET_TYPE4`가 왜 "검수"인지 같은)은 이름만으론 맥락이 부족해서, 이런 건 사람이 보완하게 남겨뒀다. **자동으로 될 건 자동, 안 될 건 사람** 이 경계를 긋는 게 설계의 절반이었다.
4. **벡터 스토어 갱신** — 위에서 만든 메타데이터를 전부 JSON으로 뽑고, 기존 ChromaDB 컬렉션을 통째로 지운 다음 새로 임베딩해서 다시 빌드한다. 증분 갱신도 고민했지만, 문서가 150개 남짓이라 통째로 다시 빌드하는 게 훨씬 단순하고 실수가 없었다. 프로토타입에선 단순한 게 정답일 때가 많다.

이 배치를 매일 새벽 크론으로 돌리게 해두면, 신규 지점이나 신규 테이블이 그다음 날 자동으로 질의 대상에 들어온다. "신규 테이블 추가 시 재빌드만으로 반영"이 목표였다.

## LLM이 자꾸 없는 컬럼을 만들어낸다

RAG로 관련 스키마를 잘 넣어줘도, 제일 골치인 게 남아 있었다. LLM이 그럴듯하게 **존재하지도 않는 컬럼을 참조하거나, JOIN을 이상하게 걸었다.** 예를 들면 ticket 테이블에 `room_group_id`가 있다고 지레짐작하고 쿼리를 짠다. 실제로는 없다. ticket은 room을 경유해야 지점을 알 수 있다. 사람이 짜도 헷갈리는 걸 LLM은 자신만만하게 틀린다.

이건 결국 프롬프트로 하나하나 잡았다.

- ERD 관계를 시스템 프롬프트에 박아뒀다 ("ticket엔 room_group_id 없음 → room 경유 필수")
- 컬럼 네이밍 규칙도 명시 ("키퍼 이름은 `display_name`, `name` 아님")
- 자주 쓰는 쿼리 패턴 6개를 예시로 넣었다. 사람들이 실제로 물어보는 질문의 정답 SQL을 통째로 보여주는 게, 규칙 열 줄보다 잘 먹혔다.
- soft delete 걸린 테이블은 항상 `WHERE deleted_at IS NULL` 붙이게 했다. 안 그러면 삭제된 데이터까지 세서 숫자가 틀린다.

그리고 온도를 두 단계로 나눴다. **SQL 생성은 temperature 0, 결과 요약은 0.3.** SQL은 같은 질문에 매번 똑같이 나와야 하니까 결정적으로, 요약은 좀 자연스러워야 하니까 살짝 풀어준 거다. 같은 질문에 어제와 오늘 다른 쿼리가 나오면 아무도 못 믿는다.

## 안전장치 — 운영 DB에 붙는다는 게 계속 걸렸다

이 시스템은 운영 DB에 직접 붙는다. LLM이 실수로 DELETE나 UPDATE를 만들면 큰일이다. 그래서 겁먹은 상태로 방어막을 쳤다.

- **SELECT 쿼리만** 실행 허용. 생성된 SQL을 파싱해서 SELECT가 아니면 아예 실행 자체를 막았다.
- LLM한테 넘기는 결과는 **최대 10행까지만**. 결과가 수천 행이면 토큰도 문제고, LLM이 그걸 요약하다가 없는 숫자를 지어낸다. 환각 방지 겸 비용 방지다.
- 모든 질의를 로그로 남김. 누가 뭘 물었고 어떤 SQL이 나갔는지 다 기록해서, 나중에 문제 생기면 추적할 수 있게 했다.

## 그래서, 왜 멈췄나

프로토타입은 잘 됐다. 데모하면 다들 신기해했다. 근데 만들고 나서, 위 안전장치를 다 붙였는데도 마음에 걸리는 게 하나 남았다.

**운영 DB에 직접 자연어 쿼리를 꽂는 구조 자체가 위험하다.** SELECT만 허용해도 그렇다. 누가 "전체 지점 지난 1년 업무 다 보여줘" 같은 무거운 질문을 아무 생각 없이 자연어로 던지면, 그게 그대로 운영 DB에 풀스캔으로 꽂힌다. SQL을 아는 개발자라면 "이건 무거우니 조심"이 되는데, 자연어로 물어보는 사람은 그 무게를 알 방법이 없다. 편하게 만들수록 위험한 질문을 쉽게 던지게 된다. 확장할수록 이 리스크가 커진다. 안전장치 몇 개로 덮을 문제가 아니라, 붙는 위치가 틀린 거였다.

그래서 방향을 틀었다. 지금은 원장 DB를 Kafka(CDC)로 BigQuery에 복제해두고, **읽기 전용인 BigQuery 위에서** 자연어 질의를 하는 쪽으로 다시 만들고 있다. 운영 DB는 안 건드리고, 분석에 최적화되고 부하가 격리된 곳에서 안전하게 조회하는 그림이다. 무거운 쿼리가 들어와도 운영 서비스는 멀쩡하다. 여기까지 오니까 앞에서 만든 RAG 구조랑 메타데이터 동기화 배치는 거의 그대로 재활용할 수 있었다. 붙는 DB만 갈아 끼우면 되는 거였다.

돌이켜보면 이 프로토타입의 제일 큰 수확은 "자연어 질의가 된다"는 확인이 아니라, **"운영 DB에 직접 붙으면 안 되겠다"는 깨달음**이었다. 되는 걸 만드는 것보다, 어디에 붙이면 안 되는지를 아는 게 더 값진 결과일 때가 있다. 다음 글에서 BigQuery 버전을 정리해보려 한다.
