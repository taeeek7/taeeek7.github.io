## 먼저, 없애고 싶었던 건 화면이 아니라 사람이 낀 루프였다

키퍼(현장 청소 스태프)는 매주 다음 주에 며칠 나올 수 있는지를 미리 알려줘야 한다. 이 가용성 정보가 있어야 그 주 청소 배정이 돌아간다. 근데 등록 방식이 이랬다. 키퍼가 관리자한테 **카톡으로** "저 다음 주 월·화·목 돼요"라고 보낸다. 관리자가 그걸 읽고 운영 시스템에 **손으로** 옮겨 적는다.

문제는 화면이 없다는 게 아니었다. **사람이 중간에 낀 것 자체**였다. 관리자 한 명이 수십 명의 카톡을 받아서 하나씩 타이핑한다. 느리고, 오타가 나고, 누구 건 빠뜨린다. 키퍼가 늘어날수록 이 관리자 한 명이 병목이 된다. 게다가 카톡 메시지는 "월화목", "월,화,목요일", "담주 3일(월화목)" 처럼 사람마다 제각각이라, 옮겨 적는 사람이 매번 해석을 해야 했다.

그래서 목표를 이렇게 잡았다. **키퍼가 직접 폰으로 등록해서 그 값이 곧장 운영 시스템에 꽂히게 한다.** 관리자라는 중계자를 통째로 빼는 거다. 이렇게 문제를 "화면 하나 만들기"가 아니라 "루프에서 사람 빼기"로 정의하고 나니, 만들 것의 범위가 자동으로 정해졌다. 화려한 UI가 아니라, **마찰 없는 입력 → 즉시 반영**만 되면 된다.

그리고 여기서 결정 하나. 이걸 React로 만들까?

## 왜 바닐라 JS였나

안 썼다. 결론부터 말하면, 이 앱이 다루는 상태(state)라고는 **체크박스 7개**가 전부였다.

React가 값을 하는 지점은 상태가 복잡할 때다. 여러 컴포넌트가 같은 데이터를 공유하고, 그 데이터가 시시각각 바뀌고, 그때마다 DOM을 어디를 어떻게 갱신할지가 골치아플 때. React의 재조정(reconciliation)은 그 골치를 대신 풀어주는 대가로 빌드 도구·번들러·의존성 트리를 데려온다. 근데 이 앱엔 그 골치가 없었다. 체크박스 7개의 상태는 브라우저 DOM이 이미 들고 있고, 제출할 때 한 번 훑어 모으면 그만이다. **React가 풀어줄 문제가 애초에 없는데 비용만 지불하는 꼴**이라, 안 쓰는 게 맞았다.

대신 프레임워크가 없다고 코드를 한 파일에 다 때려박진 않았다. 손으로 얇은 레이어를 나눴다.

- `services.js` — 백엔드 API 통신 (토큰 검증, 일정 등록)
- `main.js` — 이벤트 핸들링, 폼 제출 로직
- `uiController.js` — DOM 조작, 날짜 계산, 화면 렌더링
- `style.css` — 모바일 스타일

ES6 모듈(`import`/`export`)로 나눴고, 번들러 없이 브라우저가 모듈을 직접 읽게 했다. "통신하는 놈 / 사용자 입력 받는 놈 / 화면 그리는 놈"을 갈라둔, 손으로 짠 미니 MVC다. 프레임워크가 강제하지 않아도 이 정도 분리는 해둬야, 나중에 API 응답 형식이 바뀌었을 때 `services.js`만 열면 됐다. **프레임워크를 안 쓰는 것과 구조를 안 짜는 건 다른 얘기다.**

결과적으로 전체 **699줄**로 끝났다. 의존성이 0개라 빌드 시간도 0초, npm 취약점 경고도 0개. 정적 파일이라 Netlify에 그냥 올리면 서버 비용도 0원, CDN이 전 세계로 뿌려준다. 작은 도구엔 이게 맞다.

## 로그인 없는 인증 — 제약에서 거꾸로 설계했다

가장 고민한 게 인증이었다. 여기서도 출발점은 기술이 아니라 **현장 제약**이었다.

키퍼한테 "앱 깔고 회원가입하고 비밀번호 만드세요"는 현실적으로 안 된다. 현장 스태프고, 폰을 잠깐 꺼내 30초 만에 끝내야 하는 일이고, 일부는 한국어가 모국어가 아니다. 회원가입 폼과 비밀번호 찾기 플로우를 붙이는 순간, 그걸 못 넘어서 "그냥 관리자한테 카톡 보낼래요"로 돌아간다. 그러면 **없애려던 루프가 그대로 살아난다.** 인증의 마찰이 곧 프로젝트의 실패였다.

그래서 인증을 이렇게 뒤집었다. 로그인 화면 자체를 없앤다. 대신 키퍼마다 고유한 **암호화 토큰이 박힌 개인 링크**를 발급한다.

```
https://{서비스도메인}/?token=xxxxx
```

이 링크를 키퍼가 이미 쓰고 있는 채널(카톡)으로 보내준다. 키퍼는 링크를 누르는 순간 인증이 끝난 상태로 등록 화면에 들어와 있다. 서버 쪽은 이렇게 돈다.

```
링크 진입 → URL의 token 추출
  → POST /GetScheduleToken 으로 서버에 넘김
  → 서버가 Fernet으로 토큰 복호화 → 키퍼 식별자 매핑
  → 유효하면 등록 폼 표시 / 무효하면 에러 화면
```

토큰은 서버가 대칭키로 암호화(Fernet)해서 발급한 값이라, 클라이언트는 안을 볼 수도 위조할 수도 없다. 프론트는 그냥 URL에서 꺼내 서버로 넘길 뿐, 복호화도 검증도 전부 백엔드가 한다. 프론트에 인증 로직이 하나도 없다는 게 핵심이다.

솔직히 이 설계엔 대가가 있다. **링크 자체가 곧 신분증**이라, 그 링크를 받은 사람이 곧 그 키퍼가 된다. 링크가 유출되면 남이 대신 등록할 수 있다. 진짜 로그인이라면 막았을 구멍이다. 근데 이 앱이 다루는 건 "다음 주에 며칠 나올 수 있냐"라는, 남이 장난쳐봤자 큰일 안 나는 저위험 정보였다. 그래서 **보안 강도를 낮추는 대신 마찰을 0으로** 가져가는 트레이드오프를 의식적으로 택했다. 다루는 데이터가 민감했다면 이 설계는 그대로 못 쓴다. 인증 설계는 늘 이렇게 "무엇을 지키느냐"에서 거꾸로 정해야 한다고 다시 배웠다.

## 현장은 데스크톱이 아니라 폰이다

키퍼는 사무실 모니터 앞이 아니라 현장에서, 서서, 폰으로 이걸 연다. 이 사용 환경이 UI의 모든 결정을 지배했다.

- **모바일 퍼스트.** max-width 500px로 폰 화면에 맞췄다. 데스크톱을 먼저 그리고 줄이는 게 아니라, 처음부터 폰이 기준이었다.
- **손가락으로 누르는 UI.** 체크박스와 버튼을 크게. 마우스 커서가 아니라 엄지손가락으로 정확히 누를 수 있는 크기.
- **색으로 구분.** 근무 등록은 파란색, 휴무는 빨간색. 글자를 안 읽어도 색으로 구분되게.
- **한국어/영어 병기.** 키퍼 중엔 한국어가 익숙지 않은 분들이 있어서, 버튼·안내·확인 문구를 전부 두 언어로 나란히 뒀다. "일정 등록하기 (Register Schedule)" 식으로.
- **실수를 한 번 잡아주기.** 잘못 눌러도 제출 전에 확인 다이얼로그로 한 번 되묻는다. 현장에서 급하게 누르다 잘못 등록하는 걸 막는 안전장치.

날짜 계산은 은근히 이 맥락에서 제일 중요했다. 화면엔 항상 "다음 주 월~일"이 떠야 하는데, 문제는 **키퍼가 폰을 흘깃 보고 그냥 체크한다**는 거다. 날짜가 한 주 밀려 있어도 검산하지 않는다. 그래서 요일 계산이 틀리면 그대로 엉뚱한 주가 등록된다. 일요일에 접속하든 수요일에 접속하든 정확히 다음 주가 떠야 했다.

```javascript
const daysUntilMonday = (8 - today.getDay()) % 7 || 7;
```

이 한 줄 맞추는 데 생각보다 테스트를 많이 했다. 특히 일요일(`getDay()`가 0)이 경계라, 여기서 "이번 주"와 "다음 주"가 뒤집히기 쉬웠다.

## 제출 모델: 매번 7일 전체를 덮어쓰기

작지만 신경 쓴 설계가 하나 더 있다. 제출할 때 **선택한 날만** 보내는 게 아니라, **7일 전체를 매번** 보낸다. 선택한 날은 `scheduled`, 안 한 날은 `canceled` 상태로 붙여서.

```
선택한 날   → { work_date, status: 'scheduled' }
안 한 날     → { work_date, status: 'canceled' }
```

이렇게 한 이유는, 키퍼가 **마음을 바꿔 다시 제출하는 경우**를 깔끔하게 처리하려고다. 선택한 날만 보내면, 처음에 월·화를 등록했다가 다시 월만 등록하면 서버는 "화요일을 뺀 건지, 그냥 화요일을 언급 안 한 건지" 알 수가 없다. 7일 전체 상태를 통째로 보내면 그 주의 최종 상태가 매 제출마다 확정되니까, 서버는 그냥 덮어쓰면 된다. 재제출해도 꼬이지 않는 멱등(idempotent)한 제출이 된다. 휴무 신청 버튼은 이 위에서 그냥 **7일 전부 `canceled`** 를 보내는 특수 케이스로 처리했다.

## 배포는 정적 사이트, 환경변수는 빌드 때 주입

배포는 Netlify 정적 호스팅으로 갔다. 근데 API 엔드포인트 같은 값은 코드에 박아두기 싫었다. 그래서 빌드 시점에 Netlify 환경변수를 읽어 `env.js`를 생성하는 작은 스크립트(`generate-env.js`)를 뒀다. SPA라 새로고침·직접 진입에서 404 안 나게 `netlify.toml`로 모든 경로를 `index.html`로 리다이렉트시켰다. 실사용 모니터링은 브라우저 RUM을 100% 샘플링으로 붙여서, 현장에서 진짜로 로딩·에러가 어떤지 보이게 했다.

솔직히 이 배포 세팅을 처음에 여러 번 헤맸다. 커밋 로그에 `test`, `test3`, `test4`, `test5` 가 줄줄이 남아 있는데, 전부 환경변수 주입이 Netlify에서 제대로 되나 확인하려고 밀어붙인 삽질의 흔적이다. 로컬에선 되는데 배포 환경에서만 값이 비는, 그 흔한 시행착오였다.

## 실패와 한계도 남겨둔다

**일요일 마감 로직에서 삽질했다.** 처음엔 "일요일엔 제출 마감"이라는 규칙을 넣으려 했는데, `getDay()`가 일요일을 0으로, 토요일을 6으로 반환하는 걸 헷갈려서 마감 요일을 6이니 7이니 하며 계속 어긋났다. 커밋 로그에 `getDay 7번으로 변경`, `일요일 제출불가 조건 삭제`가 그대로 남아 있다. 결국 이 마감 조건 자체가 운영상 애매해서 도로 지웠다. 요일 인덱스는 늘 이렇게 한 칸씩 나를 속인다.

그리고 **바닐라 JS의 천장**도 분명하다. 지금이야 화면이 폼 하나라 700줄로 끝났지만, 여기에 상태가 얽힌 화면이 몇 개만 더 붙어도 손으로 DOM을 갱신하는 방식은 금방 지저분해진다. 그 경계를 넘는 순간엔 프레임워크가 답이 맞다.

## 배운 것

"작은 문제엔 작은 도구"라는 걸 다시 확인했다. 도구를 고를 때 습관적으로 제일 크고 유명한 걸 집는데, 정작 이 앱은 프레임워크를 안 쓴 게 정답이었다.

근데 더 크게 남은 교훈은 도구 선택이 아니라 **문제 정의** 쪽이었다. 이 프로젝트의 진짜 승부는 "React냐 바닐라냐"가 아니라, **없앨 대상을 화면이 아니라 사람이 낀 루프로 본 것**과, **인증을 기술이 아니라 현장 마찰에서 거꾸로 설계한 것**이었다. 문제를 그렇게 잡고 나니 나머지 기술 결정은 거의 자동으로 따라왔다. 699줄로 현장의 손일 하나를 없앴고, 관리자가 카톡 받아 대신 입력하던 작업이 통째로 사라졌다. 코드 양보다 이게 훨씬 중요했다.
