키퍼(현장 스태프)가 다음 주 근무 가능한 날을 등록해야 하는데, 원래는 관리자한테 카톡으로 알려주면 관리자가 손으로 입력했다. 사람이 중간에 끼니까 느리고 틀린다. 그래서 **키퍼가 직접 폰으로 등록**하는 웹앱을 만들기로 했다.

여기서 결정 하나. 이걸 React로 만들까?

## 왜 바닐라 JS였나

안 썼다. 화면이라곤 "날짜 체크박스 + 제출 버튼" 하나다. 이걸 위해 React + 빌드 도구 + 번들러를 얹는 건 과했다. 그래서 **프레임워크 없이 바닐라 JS**로 갔고, 결과적으로 전체 **699줄**로 끝났다.

파일은 이 정도로 쪼갰다.

- `services.js` — API 통신 (토큰 검증, 일정 등록)
- `main.js` — 이벤트 핸들링
- `uiController.js` — DOM 조작, 날짜 계산
- `style.css` — 모바일 스타일

의존성이 0개라, 빌드 시간도 0초고 npm 취약점 경고도 없다. 정적 파일이라 Netlify에 그냥 올리면 서버 비용도 0원이다. 작은 도구엔 이게 맞다.

## 신경 쓴 디테일

**로그인 없이 인증.** 키퍼한테 "앱 깔고 회원가입하세요"는 현실적으로 안 된다. 그래서 개인 고유 토큰이 박힌 링크(`?token=...`)만 카톡으로 보내주면, 그 링크로 들어와서 바로 등록하게 했다. 백엔드에서 Fernet 토큰을 풀어서 누구인지 확인한다. 로그인 화면 자체가 없다.

**"다음 주"를 정확히.** 은근 까다로운 게 날짜였다. 일요일에 접속하든 수요일에 접속하든 항상 "다음 주 월~일"이 떠야 한다. 요일 계산을 잘못하면 이번 주가 뜨거나 2주 뒤가 뜬다.

```javascript
const daysUntilMonday = (8 - today.getDay()) % 7 || 7;
```

이 한 줄 맞추는 데 생각보다 테스트를 많이 했다.

**현장은 폰이다.** 키퍼는 데스크톱 앞이 아니라 현장에서 폰으로 연다. 그래서 max-width 500px 모바일 퍼스트, 체크박스는 손가락으로 누르기 좋게 크게, 등록은 파란색 휴가는 빨간색으로 색 코딩. 실수로 잘못 누르면 확인 다이얼로그로 한 번 잡아준다.

## 배운 것

"작은 문제엔 작은 도구"라는 걸 다시 확인했다. 도구를 고를 때 습관적으로 제일 크고 유명한 걸 집는데, 정작 이 앱은 프레임워크를 안 쓴 게 정답이었다. 699줄로 현장의 손일 하나를 없앴고, 관리자가 대신 입력하던 작업이 통째로 사라졌다.

물론 화면이 복잡해지면 바닐라 JS는 금방 한계가 온다. 근데 그 경계를 넘기 전까진, 안 쓰는 게 이득일 때가 많다.
