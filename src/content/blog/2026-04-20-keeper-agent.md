사내 정책 문서나 API 스펙, DB 스키마 등이 여기저기 흩어져 있어서 필요할 때마다 찾아보는 게 꽤 번거로운 일이었다. 그때마다 최신 버전인지 확인하는 것도 일이라서, 아예 레포지토리 하나에 문서를 모아두고 AI가 이를 조회해서 답변하게 만드는 사내 지식 서버(`Keeper Agent`)를 만들어보기로 했다. 

결과적으로 FastAPI 기반의 파이썬 MCP 서버를 구축했는데, 오늘은 처음에 정석대로 만들었던 RAG 아키텍처를 일주일 만에 갈아엎고 지금의 구조로 정착하게 된 과정을 위주로 정리해본다. 

## 초기 구축: 정석대로 RAG 도입

처음엔 문서들을 청크로 쪼개고, 임베딩해서 벡터DB에 넣고, 질문이 오면 유사도를 검색해 LLM에게 넘기는 일반적인 RAG(Retrieval-Augmented Generation) 방식을 썼다. Qdrant 벡터DB에 TEI(multilingual-e5-base) 모델까지 도커로 띄우고 Confluence 문서를 연동해서 꽤 그럴듯하게 구축했었다. 

하지만 얼마 못 가 이 거창한 구조를 다 뜯어고치게 되었다. 

## RAG 대신 파일을 통째로 넘기기 

테스트를 하다 보니 API 스펙(Swagger)이나 DB 스키마 같은 구조화된 데이터는 벡터 검색을 하는 것보다 그냥 파일 전체를 LLM에게 넘겨주고 알아서 읽게 하는 편이 훨씬 정확하고 빠르다는 걸 알게 되었다. 

그래서 청킹, 임베딩, Qdrant 같은 검색 레이어들을 과감하게 다 덜어냈다. 대신 MCP 도구(`search_policy`, `search_api_spec`, `search_schema` 등)가 `knowledge/` 폴더 안의 원본 파일(json, sql, yaml)을 그대로 읽어서 반환하고, LLM이 문맥 안에서 직접 필요한 내용을 찾아 요약하도록 바꿨다. 

이렇게 하니 엉뚱한 문서를 가져와서 환각(Hallucination)을 일으키는 문제도 없어지고 응답도 한결 빨라졌다. 데이터 크기가 적당하고 구조화되어 있다면 무조건 벡터 검색을 고집할 필요가 없다는 걸 깨달았다. (기존 RAG 코드는 나중에 문서량이 엄청 늘어날 때를 대비해 주석 처리만 해두었다.)

## 도구(Tool) 설명으로 라우팅 대체하기 

RAG 라우터를 없애니 "그럼 어떤 파일을 읽어야 할지 LLM이 어떻게 판단하나?" 라는 문제가 생겼다. 
해결책은 의외로 간단했다. MCP 도구의 `description` 항목에 조회할 수 있는 문서 목록을 통째로 명시해두는 거였다. 

LLM이 이 설명을 보고 필요한 문서를 스스로 판단해서 호출하게 했고, 필요하면 여러 도구를 병렬로 호출해서 정보를 조합하도록 했다. 
예를 들어 `search_server_enums` 도구를 처음엔 인자 없이 불러서 전체 enum 목록 요약을 보고, 상세 내용이 필요할 때만 특정 enum 이름(예: `KeeperLevel`)을 인자로 넘겨서 다시 호출하도록 2단계로 구성해 컨텍스트 낭비도 줄였다. 

## 접속 방식 변경 (진입 장벽 낮추기)

처음엔 stdio 방식으로 MCP 서버를 돌렸는데, 이렇게 하니 다른 팀원들이 쓰려면 각자 로컬에 Python과 uv를 깔고 세팅을 해야 했다. 개발자가 아닌 동료들에게는 너무 큰 진입 장벽이었다. 

그래서 누구나 쉽게 붙을 수 있도록 FastAPI를 올려 HTTP 방식으로 서버를 다시 띄웠다. Claude Desktop이나 Claude Code의 `.mcp.json` 설정 파일에 `mcp-remote` 설정과 Bearer 토큰 한 줄만 넣으면 바로 연동되도록 만들었다. 
좋은 도구를 만들어도 사람들이 쓰기 어려우면 의미가 없으니, 아키텍처나 방식의 일관성보다 사용자 접근성을 우선시하는 게 맞다고 느꼈다. 

## 문서와 실제 코드 동기화 

문서를 한 곳에 모아두고 보니, 문서에 적힌 정책이나 상태값이 실제 서버 코드와 다르게 노는 경우가 종종 발생했다. 예를 들어 정책서에는 특정 상태값이 적혀 있는데, 실제 백엔드 코드에는 없는 값일 수도 있는 것이다. 

이를 해결하기 위해 서버 코드(Kotlin)의 `@Entity`에 적힌 KDoc 주석이나 `enum class` 값을 파싱해서 지식 레포에 자동으로 동기화해 주는 파이프라인을 추가했다. 
서버 레포에 배포(PR 머지)가 일어날 때마다 GitHub Actions(`sync-keeper-agent.yml`)가 돌면서 `entities.json`이나 `enums.json` 같은 파일을 업데이트하는 자동 PR을 만들어준다. 

결과적으로 에이전트는 질문을 받으면 다음과 같이 동작한다.
1. `search_policy` 도구로 정책 텍스트 확인 
2. 정책서에 적힌 `KeeperLevel.LV3` 같은 값을 `search_server_enums` 로 실제 코드 값과 대조 검증
3. `search_entity_prefix` 도구로 데이터 관계 파악 후 최종 답변 

문서가 실제 코드를 기반으로(Grounding) 유지될 수 있도록 최소한의 동기화 장치를 마련한 셈이다. 

## 마무리 

이렇게 사내 지식 서버를 구축해두니, 내가 잘못 알고 있던 정책을 AI가 실제 코드를 근거로 "주장이 틀렸습니다" 라며 정정해줄 때 가장 큰 보람을 느꼈다. 막연하게 RAG가 무조건 좋을 줄 알았는데, 상황에 따라서는 원본 데이터를 그대로 주는 게 훨씬 낫다는 교훈도 얻었다! 
앞으로는 배포 시점에 맞춰 정책 문서가 자동으로 정리되는 기능들도 더 고도화해 볼 예정이다.
