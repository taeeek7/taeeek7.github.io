업무 요청이 이메일, 전화, 카톡, 구두로 제각각 들어왔다. 당연히 누락된다. "그거 말했잖아요"와 "저는 못 들었는데요"가 반복됐다. 요청을 **한 군데로 모으는 창구**가 필요했다.

그래서 Slack 봇을 만들었다. 봇을 멘션하면 모달 폼이 뜨고, 제출하면 DB에 저장되고 담당 채널에 게시되고, 후속 자동화까지 알아서 도는 구조. 별거 아닌 것 같지만, 이걸 제대로 엮으려니 생각보다 결정할 게 많았다.

## 왜 봇을 셋이나 만들었나

처음엔 봇 하나로 다 받으려 했다. 근데 요청을 주고받는 **이해관계자가 방향마다 달랐다.** 고객사가 본사한테 넣는 요청, 운영 파트너사가 본사한테 넣는 요청, 반대로 본사가 고객사 시설관리팀에 넣는 요청 — 이 셋은 넣는 사람도, 받는 채널도, 폼 항목도, 권한도 전부 달랐다.

이걸 봇 하나에 몰아넣으면 분기 지옥이 된다. 그래서 **이해관계자별로 봇을 분리**하기로 했다. 대신 셋이 공통으로 쓰는 유틸(Slack API 래퍼, DB 커넥션, SQS 발송)은 같은 모듈 구조로 뽑아서 봇마다 복붙했다. 멀티 테넌트를 "코드 공유 + 배포 분리"로 푼 셈이다. 이렇게 나눠두니 한 봇이 죽어도 나머지는 멀쩡했고, 고객사 봇에 함부로 본사 내부 항목이 노출될 일도 없었다. 권한 분리와 장애 격리가 공짜로 따라왔다.

## 설계의 척추: trigger_id를 요청의 평생 주민번호로

만들면서 제일 고민한 게 **"요청 하나를 어떻게 끝까지 추적하지"**였다. 접수 → 처리 → 결과 보고가 다 다른 이벤트로 들어오는데, 이걸 하나로 묶을 열쇠가 필요했다.

결론은 좀 얍삽했다. Slack이 모달 열 때 던져주는 `trigger_id`를 그대로 요청의 고유 ID로 썼다. 이 값을

- 채널에 게시하는 메시지의 context 블록에 **박아두고**,
- DB 요청 테이블에도 **컬럼으로 저장하고**,
- 모달을 넘어다닐 땐 `private_metadata`에 **실어 날랐다.**

즉 요청이 태어나는 순간의 ID를 메시지·DB·모달 세 곳에 심어서, 나중에 어느 이벤트가 들어와도 이 값 하나로 원래 요청을 찾을 수 있게 했다. 별도 세션 스토어 없이 **Slack 메시지 자체가 상태 저장소**가 된 거다. 인프라를 안 늘리려는 게으름이 만든 설계인데, 결과적으로 꽤 잘 맞아떨어졌다.

## 삽질 1: Slack의 3초 룰

만들자마자 부딪힌 게 이거다. 모달을 열려면 `trigger_id`가 필요한데, **이게 3초 안에 안 쓰면 만료**된다. 근데 우리 폼은 DB에서 카테고리·담당자 목록을 조회해서 채워야 해서, 그 조회하는 사이에 3초가 지나버렸다.

해결은 **2단계 모달**이었다.

```
1단계: 일단 "Loading..." 빈 모달을 즉시 띄운다 (trigger_id 소진)
2단계: 뜬 모달의 view_id를 붙잡아뒀다가,
       DB 조회 끝나면 views_update로 실제 내용을 갈아끼운다
```

핵심은 **1단계에서 반환된 `view.id`를 잡아두는 것**이다. trigger_id는 모달을 *여는* 데 한 번 쓰고 버리는 소모품이고, 그 다음부터는 view_id로 그 모달을 갱신한다. 사용자 입장에선 모달이 바로 뜨고 잠깐 뒤 내용이 채워진다. trigger_id 만료도 피하고 반응도 빠르다. 그리고 조회에 실패하면 조용히 죽지 않게, ephemeral 메시지로 "다시 시도해주세요"를 띄웠다. 이런 건 문서만 봐선 모르고 직접 부딪혀야 안다.

## 삽질 2: 카테고리 추가할 때마다 if-else, 그리고 알파벳 함정

카테고리가 계속 늘었다. 처음엔 `if category == 'A': ... elif category == 'B': ...` 이렇게 짰는데, 카테고리 하나 추가할 때마다 분기 코드를 고쳐야 했다. 금방 지저분해졌다.

그래서 Python의 **reflection(inspect)**으로 바꿨다. DB 카테고리 테이블에 카테고리별 인덱스(`call_index`)를 두고, 그 인덱스로 폼 빌더 클래스의 메서드를 동적으로 호출하게 했다.

```python
call_index = find_category_method(category_name)      # DB에서 인덱스 조회
methods = get_class_methods(CategoryFormat)           # 클래스 메서드 목록
modal = call_method_by_index(instance, methods, call_index)
```

이제 새 카테고리는 **DB에 한 줄 + 메서드 하나** 추가하면 끝이다. if-else를 안 건드린다.

근데 여기 함정이 하나 있었다. `inspect.getmembers`는 메서드를 **이름 알파벳 순으로** 돌려준다. 그래서 인덱스랑 메서드가 어긋나면 엉뚱한 폼이 떴다. 결국 메서드 이름을 `a_basic`, `b_attachment`, `c_certification`... 이런 식으로 **알파벳 접두어를 붙여서** 정렬 순서를 강제로 고정했다. 코드에 접두어가 왜 붙었냐고 물으면 답은 "리플렉션 순서 때문"이다. 우아하진 않지만 이게 제일 안 깨졌다. (참고로 이건 LLM 라우팅이 아니라 순수 파이썬 리플렉션이다. 가끔 "AI 라우팅이냐"고 묻는데 아니다.)

## 접수에서 끝이 아니라, 끝까지 엮기

봇의 진짜 값어치는 접수가 아니라 **생명주기 추적**이었다. 그리고 이 생명주기를 **리액션(이모지)으로 굴러가는 상태 기계**로 만들었다.

```
1. 접수: 폼 제출 → DB 저장 + 채널 게시 (메시지에 trigger_id 각인)
2. 처리: 담당자가 완료 이모지 → 봇이 스레드에 "결과 작성" 버튼 등장
3. 결과: 결과 폼 작성 → 스레드 답글 + DB 업데이트
4. 후속: SQS 큐잉 → Airflow DAG 트리거
```

2번이 재밌다. 담당자가 원 메시지에 완료 이모지를 달면 `reaction_added` 이벤트가 오는데, 이 이벤트엔 채널·타임스탬프만 있고 **요청 ID가 없다.** 그래서 봇이 `conversations_history`로 원 메시지를 다시 읽어서, 아까 context 블록에 박아뒀던 trigger_id를 **거꾸로 파싱해 꺼낸다.** 상태를 별도 DB에 안 들고 메시지에서 되읽는 방식이라, 관계없는 이모지가 달리면 파싱이 비어서 조용히 무시했다(`UnboundLocalError`를 그냥 넘겼다). 정석은 아니지만 세션 관리 없이 굴리려면 이런 트릭이 필요했다.

권한도 이 단계에서 걸었다. 결과 작성 모달은 요청을 넣은 워크스페이스 도메인이 본사일 때만 열리고, 아니면 "권한 없음" 안내 모달이 뜬다. 아무나 남의 요청에 결과를 달지 못하게 한 거다.

## SQS와 Airflow를 왜 둘 다 때렸나

접수가 끝나면 후속 자동화(알림·리포팅·동기화)가 돌아야 한다. 여기서 좀 특이하게, **SQS에 메시지를 넣고 + Airflow REST API로 DAG를 직접 트리거**하는 걸 둘 다 했다.

이유가 있다. SQS는 요청 내용을 담는 **버퍼**고, Airflow DAG는 그 큐를 비우는 **워커**다. 근데 DAG를 스케줄(예: 5분마다)로만 돌리면 접수하고도 최대 5분을 기다린다. 그래서 접수되는 순간 봇이 직접 `POST /dags/{id}/dagRuns`로 DAG를 깨워서, **큐잉과 거의 동시에 처리가 시작**되게 했다. 큐로 유실을 막고, 직접 트리거로 지연을 없앤 조합이다. 접수 창구가 자동화 파이프라인의 입구가 된 셈이다.

## 읽기 DB와 쓰기 DB를 나눈 이유 (V2)

한 봇을 V2로 갈아엎으면서 DAO 패턴을 넣고 DB를 둘로 쪼갰다.

- **읽기용 DB** — 지점·객실·카테고리 같은 잘 안 변하는 마스터 데이터. 조회만 한다.
- **쓰기용 DB** — 요청·클레임 같은 트랜잭션 데이터. 여기만 INSERT/UPDATE 한다.

마스터 데이터는 다른 시스템이 관리하는 원본을 읽어오는 거라 봇이 건드리면 안 됐고, 요청 데이터는 봇이 온전히 소유했다. 소유권이 다르니 물리적으로 분리하는 게 맞았다. 여기에 환경변수로 live/test를 자동 전환하게 해서, 테스트 봇이 실서비스 DB를 오염시키는 사고를 원천 차단했다.

## 사소하지만 중요했던 것들

- **Socket Mode.** 봇을 쿠버네티스 안에서 돌리는데 공인 IP도 인바운드 방화벽 개방도 하기 싫었다. WebSocket 기반 Socket Mode를 쓰니 봇이 밖으로 연결을 걸어서, 인그레스 없이도 돌아갔다. 재연결은 내장 클라이언트의 auto_reconnect에 그냥 위임했다.
- **에러는 반드시 사람 눈에.** 모든 핸들러를 try/except로 감싸서, 터지면 **별도 알림 채널로 에러 메시지를 쐈다.** 봇이 조용히 죽는 게 제일 무섭다. 사용자는 "제출했는데요?" 하는데 로그는 어디 있는지 모르는 상황. 그래서 예외를 삼키지 않고 채널로 밀어냈다.

## 결과

이메일·전화로 5~10분 걸리던 접수가 1분 안으로 줄었고, 무엇보다 **요청 누락이 사실상 없어졌다.** "말했다/못 들었다" 싸움이 사라진 게 제일 컸다.

솔직히 지금 다시 보면 얍삽한 구석이 많다. 상태를 Slack 메시지에서 되읽는 거나, 리플렉션 순서를 알파벳 접두어로 붙잡아둔 거나, 정석은 아니다. 규모가 더 커졌으면 상태를 제대로 된 저장소로 빼고 라우팅도 레지스트리 방식으로 바꿨어야 했다. 그래도 이 봇이 준 교훈은 분명했다. **창구를 하나로 모으고, 요청 하나에 끝까지 따라붙는 ID를 심는 것** — 이 두 개만으로도 "누락"이라는 문제는 거의 다 풀렸다. 화려한 설계보다 이게 먼저였다.
