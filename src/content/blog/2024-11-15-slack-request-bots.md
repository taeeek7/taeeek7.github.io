업무 요청이 이메일, 전화, 카톡, 구두로 제각각 들어왔다. 당연히 누락된다. "그거 말했잖아요"와 "저는 못 들었는데요"가 반복됐다. 요청을 **한 군데로 모으는 창구**가 필요했다.

그래서 Slack 봇을 만들었다. 봇을 멘션하면 모달 폼이 뜨고, 제출하면 DB에 저장되고 담당 채널에 게시되고, 후속 자동화까지 알아서 도는 구조. 이해관계자가 달라서 결국 봇 3개(고객사용/파트너용/시설관리용)가 됐다.

## 삽질 1: Slack의 3초 룰

만들자마자 부딪힌 게 이거다. Slack 모달을 열려면 `trigger_id`가 필요한데, **이게 3초 안에 안 쓰면 만료**된다. 근데 우리 폼은 DB에서 카테고리·담당자 목록을 조회해서 채워야 해서, 그 조회하는 사이에 3초가 지나버렸다.

해결은 **2단계 모달**이었다.

```
1단계: 일단 빈 모달을 즉시 띄운다 (trigger_id 소진)
2단계: DB 조회 끝나면 view_update로 실제 내용 채운다
```

사용자 입장에선 모달이 바로 뜨고 잠깐 뒤 내용이 채워진다. trigger_id 만료도 피하고 반응도 빠르다. 이런 건 문서만 봐선 모르고 직접 부딪혀야 안다.

## 삽질 2: 카테고리 추가할 때마다 if-else

카테고리가 계속 늘었다. 처음엔 `if category == 'A': ... elif category == 'B': ...` 이렇게 짰는데, 카테고리 하나 추가할 때마다 분기 코드를 고쳐야 했다. 금방 지저분해졌다.

그래서 Python의 **reflection(inspect)**으로 바꿨다. DB에 카테고리별 인덱스를 두고, 그 인덱스로 폼 빌더 클래스의 메서드를 동적으로 호출하게 했다.

```python
call_index = find_category_method(category_name)
methods = get_class_methods(CategoryFormat)
modal = call_method_by_index(instance, methods, call_index)
```

이제 새 카테고리는 **DB에 한 줄 + 메서드 하나** 추가하면 끝이다. if-else를 안 건드린다. (참고로 이건 LLM 라우팅이 아니라 순수 파이썬 리플렉션이다. 가끔 "AI 라우팅이냐"고 묻는데 아니다.)

## 접수에서 끝이 아니라, 끝까지 엮기

봇의 진짜 값어치는 접수가 아니라 **생명주기 추적**이었다.

```
1. 접수: 폼 제출 → DB 저장 + 채널 게시
2. 처리: 담당자가 이모지 리액션 → "처리 완료" 버튼 등장
3. 결과: 결과 폼 작성 → 스레드 답글 + DB 업데이트
4. 후속: SQS → Airflow DAG 트리거
```

요청 하나가 스레드로 쭉 이어지니까 "이거 처리됐나?"를 안 물어봐도 됐다. 그리고 접수되는 순간 SQS로 큐잉해서 Airflow가 후속 처리(알림·리포팅·동기화)를 받아가게 했다. 접수 창구가 자동화 파이프라인의 입구가 된 셈이다.

## 결과

이메일·전화로 5~10분 걸리던 접수가 1분 안으로 줄었고, 무엇보다 **요청 누락이 사실상 없어졌다.** "말했다/못 들었다" 싸움이 사라진 게 제일 컸다. 창구를 하나로 모으는 것만으로도 이렇게 달라진다.
