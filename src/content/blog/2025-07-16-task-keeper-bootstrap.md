운영팀이 손으로 하던 일들을 자동화하려면, 여기저기 흩어진 걸 하나로 받아줄 API 서버가 필요했다. 청소 배정, 세탁 수량 입력, 이미지 URL 발급 같은 자잘하지만 매일 반복되는 것들. 이걸 담을 사내 API 서버를 하나 만들기로 했다. 혼자서.

혼자 만든다는 건 자유롭지만 무섭기도 하다. 나중에 유지보수할 사람도 나고, 구조를 이상하게 잡으면 그 벌도 내가 받는다. 그래서 기능을 넣기 전에 **구조부터** 잡았다.

## 계층은 나눠두고 시작하자

처음엔 `main.py` 하나에 다 때려박았다. 라우트 몇 개니까. 근데 기능이 두세 개 늘어나니까 바로 후회했다. DB 붙이는 코드랑 비즈니스 로직이랑 HTTP 처리가 한 함수에 뒤엉켰다.

그래서 초반에 갈아엎고 4계층으로 나눴다.

```
Router      →  HTTP 받고 검증만
  ↓
Service     →  비즈니스 로직
  ↓
Repository  →  DB 쿼리 (DAO)
  ↓
Model       →  요청/응답 DTO (Pydantic)
```

그리고 `v1`, `v2`로 버전 폴더를 나눴다. 사실 처음엔 오버엔지니어링 같았는데, 나중에 기능 스펙이 바뀌면서 `v2`를 새로 파는 게 훨씬 깔끔했다. 혼자 하니까 "미래의 나"를 위한 최소한의 예의라고 생각했다.

## 배포는 손 안 대고 싶었다

혼자 운영하는 서버라 배포에 시간을 쓰고 싶지 않았다. 그래서 push하면 알아서 올라가게 만들었다.

```
코드 push
  → GitHub Actions (도커 빌드)
  → 컨테이너 레지스트리 push
  → 배포 레포의 이미지 태그 자동 수정 + PR 자동 머지
  → ArgoCD가 감지해서 롤아웃
```

핵심은 **배포 레포를 직접 건드리지 않는다**는 거다. Actions가 이미지 태그만 `yq`로 갈아끼워서 PR을 열고 자동 머지하면, ArgoCD가 Git 상태를 보고 알아서 맞춰준다. live/test 두 벌로 워크플로를 나눠서, test에 먼저 올려보고 live에 올렸다. 배포 끝나면 Slack으로 알림도 쐈다.

> 참고로 초반엔 배포 트리거 브랜치를 `live`로 뒀다가, 나중에 `main`으로 바꿨다. 브랜치 전략이 괜히 복잡하면 혼자 헷갈린다. 단순한 게 최고다.

## 비밀은 암호화해서 Git에 넣었다

DB 비밀번호, API 키 같은 걸 어디에 둘지가 늘 고민이다. 코드에 하드코딩은 당연히 안 되고, 그렇다고 매번 서버에 손으로 넣기도 귀찮다.

그래서 **SOPS + age**를 썼다. 설정 파일을 age 키로 암호화한 `config.enc` 형태로 **암호화된 채로 Git에 커밋**하고, 배포 때 CI가 age 개인키(GitHub Secret)로 복호화해서 실제 설정 파일을 만든다. 평문 설정 파일은 `.gitignore`로 빼서 절대 커밋되지 않게 했다.

이렇게 하면 "설정도 코드처럼 버전 관리"가 되면서 비밀은 안 샌다. 혼자 관리하는 입장에서 이게 은근 마음이 편했다.

## 환경은 변수 하나로 가른다

live랑 test는 DB가 다르다. 처음엔 배포할 때마다 설정을 바꿔주다가, 실수할 것 같아서 `ENV` 환경변수 하나로 분기하게 했다.

```python
if os.getenv("ENV") == "live":
    db = LIVE_DB_CONFIG
else:
    db = TEST_DB_CONFIG
```

도커 빌드할 때 `--build-arg ENV=live`로 넘기고, 배포 워크플로가 환경에 맞게 세팅한다. 코드는 한 벌, 환경은 변수로. 이게 되고 나니까 "test에서 되던 게 live에서 안 되는" 종류의 사고가 확 줄었다.

## 남는 이야기

거창한 건 없다. 근데 혼자 서버 하나를 밑바닥부터 세우면서 배운 게 많았다. 구조를 미리 나눠두는 것, 배포를 자동으로 돌려두는 것, 비밀을 코드처럼 다루는 것. 이 세 개만 초반에 깔아두면, 그 위에 기능을 얹는 건 오히려 쉬웠다.
