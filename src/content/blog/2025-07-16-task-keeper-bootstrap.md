운영팀이 매일 손으로 하던 일들이 있었다. 그날 출근한 키퍼(현장 청소 스태프)에게 청소할 객실을 배정하고, 세탁 업체가 보낸 린넨 수량을 입력하고, 청소·세탁 사진 URL을 발급하고. 하나하나는 별거 아닌데, 매일 반복되고, 서로 붙어 있지도 않은 잡무들이었다.

이런 걸 담을 사내 API 서버를 하나 만들기로 했다. 혼자서.

## 문제는 "기능"이 아니라 "혼자, 계속"이었다

만들 기능 목록만 보면 어렵지 않다. 진짜 문제는 두 가지였다.

하나, **앞으로 뭐가 더 붙을지 모른다.** 배정으로 시작했지만 세탁이 붙고, 이미지가 붙고, 나중엔 스크럼 봇이랑 자연어 질의까지 붙었다. 성격이 제각각인 기능이 한 서버에 계속 쌓이는 구조다. 아무 생각 없이 만들면 6개월 뒤엔 손도 못 대는 덩어리가 된다.

둘, **만드는 것도 나, 유지보수도 나.** 리뷰해줄 사람도, 구조를 잡아줄 사람도 없다. 규율이 없으면 그 벌은 온전히 미래의 내가 받는다.

그래서 기능을 넣기 전에 **구조와 규칙부터** 정했다. 혼자일수록 규칙이 필요하다는 게 이 프로젝트의 첫 결론이었다.

## 설계 규칙을 코드보다 먼저 문서로 박았다

나는 이 서버를 AI 페어프로그래밍으로 많이 짰다. 그래서 아키텍처 규칙을 아예 문서로 못 박아뒀다. AI한테도, 미래의 나한테도 똑같이 적용되는 가드레일이다. 몇 개만 옮겨보면 이렇다.

**계층 규칙 — 의존은 한 방향으로만.**

```
Router      →  API 라우팅, 요청/응답, 인증
  ↓
Service     →  비즈니스 로직, 데이터 변환, 외부 API 호출
  ↓
Repository  →  DB CRUD (DAO)
  ↓
Model       →  데이터 구조 정의, 유효성 검증 (Pydantic)
```

> "Router → Service → Repository 순서로만 의존 / 계층 간 순환 참조 금지 / util 모듈은 모든 계층에서 사용 가능"

**네이밍 규칙 — 도메인마다 파일 4종 세트.** 새 기능은 예외 없이 `{도메인}_dto.py` / `{도메인}_dao.py` / `{도메인}_service.py` / 라우터 `{도메인}.py`로 떨어진다. 규칙이 이러니까 새 기능을 붙일 때 "어디에 뭘 만들지" 고민이 사라졌다. 손이 기계적으로 움직인다.

**도메인 규칙 — 언어까지 정했다.** 변수·함수명은 영어 snake_case, 근데 비즈니스 용어(하우스키퍼·객실·티켓)랑 Pydantic 필드 설명은 한국어로. 혼자 짜도 도메인 용어가 흔들리면 나중에 내가 헷갈린다.

이걸 미리 써두니까, 기능 10개가 붙어도 서버가 안 무너졌다. 규칙이 곧 뼈대였다.

## 버전은 prefix가 아니라 디렉토리로 갈랐다

API 버저닝을 라우터 prefix(`/v1/...`)로만 하는 방법도 있다. 근데 나는 **모든 계층 밑에 `v1/`, `v2/` 디렉토리를 통째로** 두는 방식을 골랐다.

이유는 하나다. 스키마가 깨지는 변경이 왔을 때, `v1`을 건드리지 않고 `v2` 트리를 통째로 새로 파서 병렬로 굴리고 싶었다. 실제로 초기 배정 API는 `v1`에, 나중에 만든 이미지 URL API는 `v2`에 깔끔하게 공존한다. 라우터만 나눴으면 서비스·DAO가 뒤엉켰을 거다. 디렉토리로 가르니까 버전이 물리적으로 격리됐다.

## 배포는 손 안 대고 싶었다

혼자 운영하는 서버라 배포에 시간을 쓰고 싶지 않았다. 그래서 push하면 알아서 올라가게 만들었다.

```
코드 push
  → GitHub Actions (도커 빌드)
  → 컨테이너 레지스트리 push
  → 배포 레포의 이미지 태그 자동 수정 + PR 자동 머지
  → ArgoCD가 감지해서 롤아웃
```

핵심은 **배포 레포를 직접 건드리지 않는다**는 거다. Actions가 이미지 태그만 `yq`로 갈아끼워서 PR을 열고 자동 머지하면, ArgoCD가 Git 상태를 보고 알아서 맞춰준다. live/test 두 벌로 워크플로를 나눠서, test에 먼저 올려보고 live에 올렸다. 배포 끝나면 Slack으로 알림도 쐈다.

이미지 태그는 `날짜+커밋SHA`로 박아서, 지금 떠 있는 게 어느 커밋인지 태그만 봐도 알게 했다. **브랜치가 곧 환경**이다 — `main`이면 live, `test`면 test.

근데 이 배포 전략은 한 번에 정해진 게 아니다. 처음엔 트리거 브랜치를 `live`로 뒀다가 나중에 `main`으로 바꿨고, test 배포도 `test` 브랜치 push에서 `test-*` 태그 push로 바꿨다. 브랜치 push는 실수로 배포가 나가기 너무 쉬웠기 때문이다. 태그로 바꾸니까 "일부러 태그를 달아야만" 배포가 나간다. 이런 건 해보기 전엔 모른다. 몇 번 데여보고 고쳤다.

## 비밀은 암호화해서 Git에 넣었다

이 서버는 운영 DB 비밀번호, LLM API 키, 스토리지 키를 들고 있다. 근데 설정도 코드처럼 버전 관리하고 싶었다. 매번 서버에 손으로 넣는 건 혼자 관리엔 오히려 위험하다.

그래서 **SOPS + age**를 썼다. 설정을 age 키로 암호화한 파일로 **암호화된 채 Git에 커밋**하고, 배포 때 CI가 개인키로 복호화해서 실제 설정을 만든다. 평문 설정 파일은 `.gitignore`로 확실히 빼서 절대 안 올라가게 했다.

이것도 운영하면서 한 번 홍역을 치렀다. age 키를 새로 발급해서 전부 재암호화하고, 복호화 스크립트가 로컬 키 파일이 있으면 그걸 쓰고 없으면 CI 시크릿으로 떨어지게 분기를 다듬었다. 키 관리는 처음 붙일 때보다 **바꿀 때** 진짜 일이 된다는 걸 그때 배웠다.

## 환경은 변수 하나로 가른다

처음엔 live로 전환할 때 설정을 손으로 바꿨다(커밋 로그에 `live DB 전환`이라는 부끄러운 흔적이 있다). 이건 실수하기 딱 좋아서, `ENV` 환경변수 하나로 분기하게 갈아엎었다.

```python
DB_CONFIG = LIVE_DB_CONFIG if os.getenv("ENV") == "live" else TEST_DB_CONFIG
```

도커 빌드할 때 `--build-arg ENV=live`로 넘기고, 배포 워크플로가 환경에 맞게 세팅한다. 코드는 한 벌, 환경은 변수로. 이게 되고 나서 "test에서 되던 게 live에서 안 되는" 종류의 사고가 확 줄었다. 환경을 일급 시민으로 끌어올린 셈이다.

## 솔직한 부채: 내가 쓴 규칙을 내가 다 못 지켰다

정직하게 남긴다. 나는 DB 규칙 문서에 "**SQL 인젝션 방지를 위한 파라미터화 쿼리 사용**", "연결 풀링 구현", AI 규칙 문서에 "동시 요청 수 제한 / 캐싱"이라고 써놨다. 근데 배정 쿼리 일부는 급하게 f-string으로 값을 박아 넣었고(내부용이라 방심했다), 커넥션 풀링도 캐싱도 아직 안 붙였다. 세탁·이미지 쪽 DAO는 규칙대로 파라미터 바인딩을 했는데, 배정 쪽은 못 지킨 거다.

혼자 짜면 이런 부채가 조용히 쌓인다. 규칙을 문서로 박아둔 덕에, 적어도 **뭘 못 지켰는지는 정확히 안다**. 그게 규칙을 먼저 쓴 두 번째 이득이었다.

거창한 결론은 없다. 근데 혼자 서버 하나를 밑바닥부터 세우면서 확실히 배운 게 있다. **기능보다 규칙을, 배포보다 파이프라인을, 비밀보다 비밀을 다루는 방식을 먼저 깐다.** 이 세 개만 초반에 박아두면, 그 위에 기능을 얹는 건 오히려 쉬웠다.
