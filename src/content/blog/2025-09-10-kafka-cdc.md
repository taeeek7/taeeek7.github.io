Airflow로 배치 ETL을 돌리다 보니 한계가 보였다. 아무리 자주 돌려도 배치는 배치다. 운영 DB가 방금 바뀐 걸 분석계에서 보려면 다음 배치까지 기다려야 한다. 로그 분석 환경을 제대로 만들려니 이 지연이 걸렸다.

그래서 **CDC(Change Data Capture)**를 붙였다.

## binlog를 읽어서 실시간으로 전파

Debezium MySQL 커넥터를 Kafka Connect 위에 올렸다. 이게 MySQL의 binlog를 읽어서 INSERT/UPDATE/DELETE를 실시간으로 감지하고, Kafka 토픽으로 쏴준다.

```
MySQL (운영 DB)
   ↓ Debezium CDC (binlog 실시간)
Kafka Topics
   ↓
GCS / BigQuery / OpenSearch (동시에)
```

배치로 수 시간 걸리던 데이터 지연이 **수 초**로 줄었다. 운영 DB의 변경이 거의 바로 분석계에 반영된다.

## 하나의 변경을 세 군데로 동시에

CDC의 진짜 맛은 **멀티 싱크**였다. 같은 변경 이벤트 하나를 용도가 다른 세 곳에 동시에 흘려보냈다.

| Sink | 포맷 | 용도 |
|------|------|------|
| GCS | JSONL + gzip | 로그 아카이빙 (날짜 계층) |
| BigQuery | 자동 스키마 | 실시간 분석 쿼리 |
| OpenSearch | JSON | 실시간 로그 검색 |

중요한 건 **각 싱크를 독립으로 운영**한 거다. OpenSearch가 죽어도 BigQuery 적재는 멀쩡하게 돈다. 하나 터졌다고 파이프라인 전체가 멈추면 안 되니까.

## 귀찮았던 것들

솔직히 CDC는 붙이고 나면 좋은데 세팅이 손이 많이 간다.

- **커넥터 관리**: 처음엔 curl로 커넥터를 등록했는데 환경(local/test/live)마다 13~14개씩 되니까 금방 지옥이 됐다. 결국 Python REST API 클라이언트를 만들어서 JSON 설정 파일로 일괄 배포하게 했다.
- **시크릿**: DB 비번, GCP 서비스 계정 키를 Git에 그냥 둘 순 없어서 SOPS + Age로 암호화해서 커밋하고, CI에서 복호화해 이미지에 넣었다.
- **환경 격리**: 오프셋이랑 Storage Topic을 환경별로 분리 안 하면 테스트가 운영을 건드린다. 이거 한 번 꼬이면 디버깅이 정말 괴롭다.

## 정리

CDC를 붙이고 나서 데이터 파이프라인의 사고방식이 좀 바뀌었다. "언제 당겨올까(배치 주기)"를 고민하던 게 "바뀌면 바로 흐른다"로 바뀌니까, 실시간 알림이든 검색이든 그 위에 얹기가 훨씬 쉬워졌다. 대신 관리할 부품이 늘어난 건 감수해야 한다.
