2년쯤 전에 [우당탕탕 Airflow 도입기](/blog/2024-10-07-airflow-intro/)를 쓴 적이 있다. docker-compose로 겨우 웹서버 하나 띄우고 좋아하던 시절이었다. 그때는 DAG이 몇 개 없었는데, 지금 세어보니 **117개**가 됐다. 그동안 뭘 자동화해왔는지, 117개나 되는 걸 어떻게 정리하고 있는지 한번 정리해본다.

## 진짜 없애고 싶었던 건 "사람이 까먹는 것"이었다

제일 먼저 없애고 싶었던 건 **매일 아침 반복하던 손일**이었다. 운영팀이 매일 구글 시트 대시보드를 손으로 갱신하고 있었다. 배정 현황, 청소 현황, 수입 명세서… 하루에 1~2시간씩 갔다.

근데 손일의 진짜 문제는 시간이 아니었다. **언젠가 빠뜨린다는 것**이다. 새벽에, 주말에, 담당자가 휴가일 때. 사람이 매일 하는 일을 100% 빠짐없이 하는 건 구조적으로 불가능하다. 운영 이슈도 마찬가지였다. 청소가 밀렸거나 클레임이 들어온 걸 몇 시간 뒤에야 알면, 그건 이미 늦은 거다. 그래서 문제를 이렇게 다시 정의했다. **일회성으로 사람이 챙기던 일을, 스케줄러가 빠짐없이 챙기는 일로 바꾼다.** 이게 117개 전체를 관통하는 목표가 됐다.

## 시작은 리포팅이었다

첫 타깃은 그 구글 시트였다. MySQL 운영 DB에서 데이터를 뽑아서 → 가공하고 → 구글 시트에 자동으로 꽂는 DAG. 10개쯤 만들고 나니 **매일 10시간 넘던 팀 리소스가 사라졌다.** 사람이 하던 걸 스케줄러한테 넘기니 새벽에도 알아서 돌았다.

시트 조작이 은근 지저분한 일이라, 셀 범위 갱신이나 포맷 같은 걸 매번 새로 짜지 않으려고 래퍼 모듈로 빼뒀다. 이게 나중에 "복붙하지 않기"의 시작이 됐는데, 그 얘긴 뒤에서 하겠다.

## 그다음은 데이터 웨어하우스

리포팅을 자동화하다 보니 자연스럽게 다음 문제가 보였다. 운영 DB(MySQL)를 분석용으로 직접 쿼리하는 건 위험하고 느리다. 무거운 집계 쿼리 하나가 운영 DB를 물고 늘어지면 실서비스가 느려진다. 그래서 **MySQL → BigQuery ETL**을 붙여서 분석 부하를 운영 DB에서 떼어냈다.

핵심은 적재 방식이었다. 그냥 타깃 테이블을 지우고 새로 넣으면, 중간에 실패했을 때 데이터가 통째로 날아간다. 그래서 **Staging → MERGE** 패턴을 표준으로 박았다.

- 먼저 Staging 데이터셋에 적재하고
- 검증이 끝나면 Main 테이블에 MERGE(변경분만 반영)
- 적재가 중간에 깨져도 Main은 멀쩡하게 유지

이 패턴이 은근 중요했다. 타깃을 직접 TRUNCATE하지 않으니, 실패해도 무결성이 지켜지고 마음 편히 매일 돌릴 수 있었다. 지금은 20개 넘는 데이터 마트가 일/시간 단위로 자동 갱신된다.

서버 로그도 골칫거리였다. JSON.gz로 쌓이던 대용량 로그를 그대로 BigQuery에 쿼리하면 느리고 비쌌다. 그래서 GCS의 JSON.gz를 스트리밍으로 읽어서 **Parquet으로 변환**해 넣었더니, 쿼리 비용이랑 속도가 절반 이하로 줄었다. Columnar 포맷의 힘이다. 파일을 통째로 메모리에 올리지 않고 라인 단위로 흘려보내게 짠 게 포인트였는데, 안 그러면 큰 로그에서 워커가 메모리로 죽는다. (이건 실제로 한 번 죽고 나서 배웠다.)

## 117개를 어떻게 정리하나

여기서부터가 이 글의 본론이다. DAG이 20개를 넘어가면서, 하나하나 잘 도는 것보다 **전체를 어떻게 정리하느냐**가 더 큰 문제가 됐다. 몇 가지 규칙으로 눌렀다.

**1. 이름만 보고 뭐 하는 DAG인지 알게 한다.** 파일명·dag_id에 역할 접두사를 붙였다. `bq_*`는 BigQuery 적재, `alert_*`는 감지·알림, `api_*`는 외부 API 폴링, `transform_*`은 로그 변환, `match_*`는 대조·정산. 117개가 UI에 쭉 뜰 때, 접두사만 봐도 성격이 갈린다. 별거 아닌 것 같아도 이게 없으면 몇 달 뒤의 내가 내 DAG을 못 찾는다.

**2. 코드는 한 벌, 환경은 파라미터로.** test 서버랑 live 서버에 DAG을 따로 두지 않았다. 코드 맨 위에서 환경 변수(`PROPERTY_ENV`)를 Airflow Variable로 읽고, dag_id를 `{env}_...` 형태로 찍는다. 같은 파일이 test에선 test용으로, live에선 live용으로 동작한다. Slack 알림 채널이나 도메인 URL, 접속할 DB 커넥션도 이 값 하나로 갈린다. 환경마다 코드를 복제하는 순간 두 벌이 어긋나기 시작하는데, 그걸 원천 차단한 거다.

**3. 실행은 CeleryExecutor로 흩뿌린다.** 117개가 시간대별로 겹쳐 돌기 때문에, 한 프로세스에서 순차로 처리하면 밀린다. CeleryExecutor로 워커를 여러 개 두고 최대 32 태스크까지 동시에 돌게 했다. 스케줄러는 트리거만 하고 실제 무거운 적재는 워커가 나눠 가진다. 무거운 ETL DAG이 가벼운 알림 DAG을 막지 않는 게 중요했다.

## 복붙하지 않기 — 유틸 모듈과 범용 DAG

117개를 만들면서 제일 크게 배운 건 이거다. **비슷한 DAG을 복붙하기 시작하면 진다.** 복붙한 순간엔 빠른데, 공통 로직에 버그가 하나 생기면 스무 군데를 다 고쳐야 한다.

그래서 공통 로직을 유틸 모듈로 뺐다. BigQuery 적재, Slack 알림, MySQL 쿼리·배치 INSERT, 외부 API 호출, 시트 조작 같은 걸 9개쯤 모듈로 정리했다. 새 DAG은 이제 "무슨 데이터를, 어디서 어디로, 언제"만 쓰면 되고 적재의 세부는 모듈이 처리한다. 새 DAG 만드는 시간이 체감 절반 이하로 줄었다. 처음엔 복붙하다가 결국 모듈로 뺐는데, 진작 할 걸 그랬다.

여기서 한 발 더 나갔다. 초기 백필처럼 **테이블만 다르고 로직은 똑같은** 작업은, DAG을 테이블마다 만드는 대신 **하나의 범용 DAG에 파라미터로 넘기게** 했다. 트리거할 때 대상 테이블과 환경을 실행 인자(`--conf`)로 주면, 컬럼 목록이랑 타입은 코드에 박지 않고 **타깃 스키마에서 런타임에 자동으로 뽑아온다.** 그래서 테이블이 새로 늘어도 코드를 안 건드려도 된다. DAG 하나가 수십 개 몫을 하는 셈이다. "DAG을 늘리지 않고 늘리는" 이 방향이, 117개에서 더 안 터지게 버티는 방법이라고 생각한다.

물론 파라미터를 받는 만큼 방어도 필요했다. 테이블명은 정규식으로 식별자 형식을 강제해서 SQL 인젝션을 막고, 환경 값은 화이트리스트로만 받는다. 잘못된 값이 오면 태스크를 아예 실패시킨다. 범용으로 만들수록 입력을 안 믿는 게 원칙이다.

## 알림 DAG은 "감지"와 "실패"를 분리했다

운영 이슈는 늦게 알수록 손해다. 청소 상태, 고객 클레임, 분실물, 예약 취소 같은 걸 감지하는 알림 DAG을 14개 만들어서 Slack으로 바로 쏘게 했다. 수 시간 걸리던 이슈 감지가 수 분으로 줄었다.

알림을 만들면서 신경 쓴 건 **두 종류의 신호를 섞지 않는 것**이었다.

- **감지된 이슈** (예: 클레임이 들어옴, 적재가 정체됨) → 담당자가 보는 전용 채널로 발송하고, DAG 자체는 정상 종료.
- **DAG의 실패** (코드 예외, BigQuery 오류, Slack 발송 실패) → 실패 콜백으로 개발 알림 채널에 따로.

이 둘을 한 채널에 섞으면, 진짜 운영 이슈랑 파이프라인 버그가 뒤엉켜서 둘 다 무뎌진다. 채널을 나누니까 "이건 운영팀이 대응할 것 / 이건 내가 고칠 것"이 명확해졌다. 실패 알림은 DAG이 도는 서버(test/live)를 기준으로 라우팅했다. 실패 로그 URL이 그 서버의 웹서버를 가리켜야 바로 눌러 들어갈 수 있으니까.

파이프라인 자체를 감시하는 헬스체크 DAG도 있다. 실시간 적재(CDC 싱크)가 죽거나 밀리는 걸 주기적으로 확인하는데, 두 개의 신호를 본다. (1) 커넥터·태스크 상태가 FAILED나 OOM인지, (2) 태스크가 살아있다(RUNNING)고 표시돼도 실제로 **마지막 적재가 언제였는지**. 두 번째가 은근 중요하다. 프로세스는 안 죽었는데 데이터만 안 흐르는 경우가 있어서, 상태만 보면 놓친다.

임계값은 감으로 정하지 않았다. 안정 구간의 실제 적재 간격을 보니 최대 10분이었고, 정상 최대의 3배쯤인 30분을 "정체" 기준으로 잡았다. 감시 주기도 처음엔 10분이었다가, 워커가 너무 자주 뜨고 지는(pod churn) 부담이 있어서 15분으로 늘렸다. 15분이어도 장애 감지는 그 안에 되니까. 그리고 감시용 REST 호출은 한 번 실패했다고 바로 알림 쏘지 않고, 짧게 3번 재시도한 뒤에도 안 되면 그때 "워커 다운"으로 알린다. 순간적인 네트워크 blip에 매번 알림이 울리면, 그 알림은 곧 무시당하기 때문이다.

## 삽질과 한계

정리하고 보니 그럴듯하지만, 삽질도 많았다.

**시간대 처리에서 하루가 밀렸다.** 적재 윈도우를 Airflow의 `ds` 매크로(UTC 기준)로 잡았는데, 우리는 전부 KST로 생각하고 있었다. UTC↔KST 9시간 차이 때문에 증분 적재가 하루씩 어긋나서, 한동안 데이터가 미묘하게 비는 걸 뒤늦게 발견했다. 스케줄러의 시간과 내 머릿속의 시간이 다르다는 걸 아프게 배웠다.

**전체 적재하다 워커가 메모리로 죽었다.** 초기 백필처럼 큰 테이블을 한 번에 DataFrame으로 올리면 OOM이 난다. 결국 정렬 키 기준으로 잘라서(keyset 배치) 나눠 넣도록 바꿨다. 대용량은 "한 번에"가 거의 항상 틀린 답이었다.

그리고 제일 솔직한 한계. **마트마다 적재 DAG을 하나씩 만드는 방식은 결국 안 늘어난다.** 새 분석 지표가 필요할 때마다 DAG을 찍어내면, 언젠가 이 117개가 200개가 되고 관리가 안 된다. 그래서 지금은 방향을 틀고 있다. 운영 DB의 변경을 실시간으로 복제(CDC)해두고, 마트는 그 위에 BigQuery 뷰나 예약 쿼리로 얹는 쪽으로. 배치 로더 DAG을 하나씩 늘리는 대신, **데이터를 한 번 잘 복제해두고 그 위에서 조합**하는 구조다. 예전에 열심히 만든 로더 DAG들을 지금 하나씩 걷어내고 있는데, 잘 만든 걸 스스로 지우는 게 데이터 파이프라인 일의 일부라는 것도 이번에 배웠다.

## 지금 와서 느끼는 것

117개를 운영하면서 배운 건, **파이프라인의 진짜 적은 "느린 쿼리"가 아니라 "사람이 까먹는 것"**이라는 거다. 손으로 하던 일은 언젠가 빠뜨린다. 그걸 스케줄러한테 넘기는 순간 그 걱정이 사라진다. 대신 이번엔 스케줄러가 조용히 틀리지 않는지를 내가 감시해야 하고, 그래서 알림과 헬스체크에 그렇게 공을 들인 거다.

그리고 규모가 커질수록 중요한 건 DAG을 잘 만드는 게 아니라, **DAG을 덜 만들어도 되게 만드는 것**이었다. 유틸 모듈, 범용 파라미터 DAG, CDC 위의 뷰. 다 "복붙과 개수 증가를 어떻게 안 하느냐"의 이야기다.

다음엔 이 파이프라인 위에서 뭘 더 얹고 있는지 — LLM 기능 정확도를 측정하는 대시보드나, BigQuery 자연어 질의 같은 걸 정리해보려 한다.
