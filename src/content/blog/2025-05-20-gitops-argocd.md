서비스가 20개쯤 되니까 배포가 일이 됐다. 이미지 빌드하고, 서버 들어가서 매니페스트 고치고, 적용하고, 잘 떴나 확인하고… 환경(live/test/staging)까지 있으니 실수가 안 날 수가 없었다. 무엇보다 무서운 건 "test에 넣을 걸 live에 넣는" 사고였다. 손으로 하는 배포는 사람이 피곤한 밤에 특히 위험했다.

그리고 이건 나 혼자 겪는 문제가 아니었다. 서비스가 20개면 배포 방식도 20개로 갈라진다. 누구는 이렇게, 누구는 저렇게 올리면, 나중에 "이 서비스는 어떻게 배포했더라"를 매번 새로 기억해내야 한다. **배포가 지식이 되면 안 된다. 배포는 시스템이어야 한다.** 그래서 문제부터 다시 정의했다. 내가 풀어야 하는 건 "배포를 빠르게"가 아니라 "**배포를 예측 가능하게, 그리고 나 없이도 굴러가게**"였다.

## 진실의 원천을 옮겼다

그래서 **GitOps**로 갔다. 핵심 아이디어는 하나다.

> **Git 저장소가 진실의 원천(Single Source of Truth)이다. 클러스터는 Git 상태를 따라간다.**

전에는 진실이 클러스터 안에 있었다. "지금 live에 뭐가 떠 있지?"의 답은 클러스터한테 물어봐야 알았고, 누가 언제 뭘 바꿨는지는 아무도 몰랐다. GitOps는 이 진실을 Git으로 끌어냈다. 이제 "지금 뭐가 떠 있어야 하는지"는 Git을 보면 된다. 내가 클러스터를 직접 안 만진다. Git을 바꾸면 ArgoCD가 알아서 클러스터를 그 상태로 맞춘다.

이 전환이 주는 세 가지가 있었다. **선언적**(원하는 상태만 적으면 도달 경로는 도구가 알아서), **감사 가능**(모든 변경이 커밋과 PR로 남으니 "누가 왜 이걸 바꿨나"가 git log에 다 있다), **자가 치유**(누가 손대서 틀어져도 Git 기준으로 되돌아온다). 손으로 배포할 땐 이 셋 중 하나도 없었다.

## 레포를 둘로 쪼갠 이유

설계에서 처음 내린 결정은 배포 저장소를 둘로 나눈 거였다.

- **서비스 배포 레포** — 우리가 만든 애플리케이션 20개. 이미지 태그가 매일 수십 번 바뀐다.
- **인프라 구성 레포** — ArgoCD 자신, Airflow, 대시보드(Redash), APM, cert-manager, 인그레스 컨트롤러 같은 판때기. 거의 안 바뀐다.

성격이 완전히 다르기 때문이다. 서비스 레포는 **변경이 잦고 자동화가 핵심**이라, 봇이 하루에도 수십 번 이미지 태그 커밋을 쌓는다(실제로 이 레포 커밋의 대부분이 자동 생성된 배포 PR이다). 반대로 인프라 레포는 **변경이 드물지만 한 번 잘못되면 클러스터 전체가 흔들리는** 곳이라, 사람이 신중하게 손대야 한다. 이 둘을 한 레포에 섞으면 자동 커밋 홍수에 진짜 중요한 인프라 변경이 파묻힌다. 그래서 나눴다.

## 파이프라인

코드 푸시부터 배포까지 이렇게 흐른다.

```
코드 푸시
  → GitHub Actions: 도커 빌드 + 컨테이너 레지스트리 푸시
  → 서비스 배포 레포의 values-{env}.yaml 이미지 태그 자동 변경
  → PR 자동 생성 + automerge 라벨 → 스쿼시 머지
  → ArgoCD가 Git 변경 감지 → Helm 렌더 → K8s 적용
  → Rolling Update + 3단계 헬스체크
  → Self-Heal (드리프트 자동 복구)
```

사람이 하는 건 코드 푸시까지다. 그 뒤론 손 안 댄다. 배포가 평균 5~10분이면 끝나고(예전 수동 배포는 30분~1시간), 무엇보다 **매번 똑같이** 된다.

여기서 은근히 중요한 게 두 가지 있다. 하나는 **이미지 태그를 `YYYYMMDD_HHMMSS-{git-sha}` 포맷으로 박은 것.** `latest`를 안 쓴 이유가 있다. `latest`는 "지금 이게 뭔지"를 감춘다. 시각+커밋 해시를 태그에 박으면, 클러스터에 떠 있는 파드만 봐도 어느 커밋인지 역추적이 된다. 롤백도 그냥 이전 태그로 되돌리면 끝이다. 다른 하나는 **automerge를 라벨로 트리거한 것.** 봇이 만든 이미지 태그 PR에 `automerge` 라벨이 붙으면 액션이 스쿼시 머지하고 브랜치를 지운다. 사람이 매번 "머지" 버튼을 누를 필요가 없다.

## ArgoCD Application 하나하나가 선언문이다

서비스마다, 환경마다 ArgoCD Application을 하나씩 만들었다(`live-keeper`, `test-keeper` 이런 식으로). 이 Application 매니페스트가 사실상 "이 서비스는 이 레포의 이 경로를, 이 values 파일로, 이 네임스페이스에 배포한다"는 선언문이다. 여기 sync 정책을 어떻게 쓰느냐가 GitOps의 실제 성격을 정한다. 내가 쓴 핵심은 이거였다.

```yaml
syncPolicy:
  automated:
    prune: true       # Git에서 지운 리소스는 클러스터에서도 삭제
    selfHeal: true    # 드리프트 자동 복구
  syncOptions:
    - Validate=true            # helm 렌더 결과 유효성 검사
    - PruneLast=true           # 다 적용한 뒤 마지막에 prune (충돌 방지)
    - ApplyOutOfSyncOnly=true  # 어긋난 리소스만 apply
  retry:
    limit: 5
    backoff: { duration: 5s, factor: 2, maxDuration: 3m }
```

`prune: true`는 은근히 무서운 옵션이다. Git에서 리소스를 지우면 클러스터에서도 진짜 지운다. 근데 이게 있어야 "Git = 진실"이 완성된다. Git에 없으면 클러스터에도 없어야 하니까. 대신 `PruneLast`로 삭제를 맨 마지막에 몰아서, 새 리소스가 다 올라온 뒤에 옛 리소스를 치우게 했다. 실패하면 지수 백오프로 다섯 번까지 재시도한다. 이 몇 줄이 "배포하다 반쯤 실패해서 클러스터가 이상해지는" 상황을 대부분 막아줬다.

## 좋았던 것 두 가지

**1. Self-Healing.** 누가 클러스터를 직접 손대서 설정이 틀어져도(드리프트), ArgoCD가 Git 기준으로 도로 돌려놓는다. "누가 라이브 만졌어?"를 걱정 안 해도 된다. Git에 없는 변경은 살아남지 못한다.

**2. 무중단 배포.** `maxUnavailable: 0`으로 두고 롤링 업데이트를 걸어서, 배포 중에도 항상 최소 대수는 살아있게 했다. 여기에 3단계 프로브(startup/liveness/readiness)와 종료 처리를 붙였다. 파드가 죽을 때 `terminationGracePeriodSeconds`(60초)보다 `preStop` sleep(50초)을 일부러 더 짧게 잡았는데, 이유가 있다. preStop이 끝나기 전에 파드가 강제로 죽으면 처리 중이던 요청이 잘린다. 그래서 "인그레스에서 이 파드를 빼는 시간 → preStop → grace period"가 순서대로 여유를 두고 흘러가게 시간 예산을 짰다. 무중단은 옵션 하나로 되는 게 아니라 이 타이밍을 맞추는 일이었다.

## 표준화가 결국 답이었다

20개 서비스를 관리하는 비결은 화려한 게 아니라 **다 똑같이 생기게 만든 것**이었다. 모든 서비스가 같은 Helm 차트 구조(deployment/service/hpa/namespace)를 따르게 했다. 공통 라벨이나 셀렉터 같은 반복되는 조각은 `_helpers.tpl` 한 군데에 몰아넣고, 서비스별로 다른 건 오직 `values-{env}.yaml`에만 두게 했다. 그러니까 새 서비스 추가가 **values 파일 하나 채우는 일**이 됐다. 이미지 경로, 레플리카 수, 리소스 한도, 헬스체크 경로만 적으면 30분이면 배포 환경이 선다.

HPA도 이 틀 안에서 values 몇 줄로 껐다 켰다 했다. 트래픽을 타는 무상태 서비스는 CPU 70% 기준으로 3~12대 사이에서 자동 스케일링, 그렇지 않은 서비스는 그냥 고정. 차트를 건드릴 일이 없으니 "이 서비스만 특별하게" 손대다가 실수할 여지도 사라졌다. 표준화의 진짜 이득은 자동화가 아니라 **틀리게 만들 방법이 없어지는 것**이었다.

## 시크릿을 Git에 넣는 법

GitOps로 가면 딱 하나 걸리는 게 있다. "그럼 DB 비밀번호나 kubeconfig 같은 시크릿은 어디 두지?" Git이 진실의 원천이면 시크릿도 Git에 있어야 하는데, 평문으로 넣을 순 없다.

그래서 **SOPS + Age**로 갔다. 시크릿이 든 values 파일을 통째로 암호화해서, 암호문 상태로 Git에 커밋한다. 원본 평문 파일은 `.gitignore`로 아예 추적에서 빼두고, 저장소엔 `.enc.yaml` 암호문만 올라간다. 복호화 키(Age 비밀키)는 Git에 절대 안 들어가고 로컬과 CI에만 둔다. 작업할 땐 스크립트 한 번으로 전체를 복호화하고, 고친 뒤 다시 암호화해서 푸시하는 흐름이다.

이게 주는 건 **시크릿도 버전 관리가 되는데 평문 유출은 막힌다**는 것. 비밀번호를 바꿔도 diff가 남으니 "언제 뭘 바꿨나"가 추적되고, 그렇다고 GitHub에서 누가 훔쳐봐도 암호문이라 쓸모가 없다. 시크릿마다 별도 볼트 시스템을 붙이는 건 20개 규모에선 과했고, "Git 하나로 다 관리한다"는 GitOps 원칙을 시크릿까지 끌고 가는 데는 이 방식이 딱 맞았다.

## 삽질과 한계

당연히 곱게 되진 않았다. 몇 개만 솔직히 적는다.

**automerge가 처음엔 무서웠다.** 사람 눈 없이 봇 PR이 자동으로 머지돼서 바로 배포로 이어지니까, 초반엔 "이러다 사고 나는 거 아냐" 싶었다. 근데 잘 생각해보면 이미지 태그만 바뀌는 PR이라 사람이 봐도 볼 게 없다. 리뷰가 의미 있는 건 코드 PR이지, 기계가 만든 태그 변경이 아니다. 검증은 CI 빌드와 헬스체크가 하는 거고, 사람 승인은 여기서 안전장치가 아니라 병목이었다. 이걸 받아들이는 데 시간이 좀 걸렸다.

**Self-Heal한테 손수정을 몇 번 되돌려 맞았다.** 급해서 클러스터를 직접 kubectl로 고쳤더니, 몇 분 뒤 ArgoCD가 Git 기준으로 조용히 원상복구해버렸다. "어? 방금 고친 게 왜 없어졌지?" 하고 한참 헤맸다. 이게 버그가 아니라 기능이라는 걸 몸으로 배웠다. GitOps에선 **긴급 수정도 Git을 거쳐야 한다.** 클러스터를 직접 만지는 습관 자체를 버려야 했다.

**초기 셋업은 러닝 커브가 꽤 있었다.** ArgoCD Application, Helm 차트, sync 옵션, SOPS 키 관리를 다 이해하고 엮는 데 시간이 걸렸다. 손으로 배포하면 오늘 당장 되지만, GitOps는 판을 까는 데 며칠이 든다. 그리고 레포엔 아직 예전 방식의 흔적(`legacy/`)이 남아 있다. 한 번에 갈아엎은 게 아니라 서비스 하나씩 옮기면서 마이그레이션한 자국이다. 삽질의 화석이다.

## 배운 것

GitOps로 오면서 사고방식이 바뀌었다. "내가 어떻게 배포하지"가 아니라 "**원하는 상태를 Git에 어떻게 적어두지**"로. 배포는 명령이 아니라 선언이 됐다. 그리고 손으로 하던 걸 시스템한테 넘길 때마다, 사람이 실수할 여지가 하나씩 사라졌다. 인프라도 결국 "사람이 까먹고 실수하는 것"을 없애는 게 절반이다.

제일 크게 남은 건 이거다. 좋은 인프라는 나를 유능하게 만드는 게 아니라, **내가 없어도 굴러가게** 만든다. 배포가 내 머릿속 지식이 아니라 저장소 안의 선언이 됐을 때, 비로소 나는 배포에서 손을 뗄 수 있었다.
