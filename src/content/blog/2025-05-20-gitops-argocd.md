서비스가 20개쯤 되니까 배포가 일이 됐다. 이미지 빌드하고, 서버 들어가서 매니페스트 고치고, 적용하고, 잘 떴나 확인하고… 환경(live/test/staging)까지 있으니 실수가 안 날 수가 없었다. "test에 넣을 걸 live에 넣는" 사고가 무서웠다.

그래서 **GitOps**로 갔다. 핵심 아이디어는 하나다.

> **Git 저장소가 진실의 원천(Single Source of Truth)이다. 클러스터는 Git 상태를 따라간다.**

내가 클러스터를 직접 안 만진다. Git을 바꾸면 ArgoCD가 알아서 클러스터를 그 상태로 맞춘다.

## 파이프라인

코드 푸시부터 배포까지 이렇게 흐른다.

```
코드 푸시
  → GitHub Actions: 도커 빌드 + NCR 푸시
  → argocd-deploy 레포의 values.yaml 이미지 태그 자동 변경
  → PR 자동 생성 + automerge
  → ArgoCD가 Git 변경 감지 → Helm 렌더 → K8s 적용
  → Rolling Update + 헬스체크
```

사람이 하는 건 코드 푸시까지다. 그 뒤론 손 안 댄다. 배포가 평균 5~10분이면 끝나고(예전 수동 배포는 30분~1시간), 무엇보다 **매번 똑같이** 된다.

## 좋았던 것 두 가지

**1. Self-Healing.** 누가 클러스터를 직접 손대서 설정이 틀어져도(드리프트), ArgoCD가 Git 기준으로 도로 돌려놓는다. "누가 라이브 만졌어?"를 걱정 안 해도 된다. Git에 없는 변경은 살아남지 못한다.

**2. 무중단 배포.** `maxUnavailable: 0`으로 두고 롤링 업데이트 + 3단계 프로브(startup/liveness/readiness)를 걸어서, 배포 중에 서비스가 죽는 일이 없게 했다.

## 표준화가 결국 답이었다

20개 서비스를 관리하는 비결은 화려한 게 아니라 **다 똑같이 생기게 만든 것**이었다. 모든 서비스가 같은 Helm 차트 구조(deployment/service/hpa/namespace)를 따르게 했다. 그러니까 새 서비스 추가가 values 파일 하나 채우는 일이 됐다. 30분이면 배포 환경이 선다.

시크릿은 SOPS + Age로 암호화해서 Git에 넣었다. 이러면 시크릿도 버전 관리가 되면서 평문 유출은 막힌다. CI에서 키로 복호화해서 쓴다.

## 배운 것

GitOps로 오면서 사고방식이 바뀌었다. "내가 어떻게 배포하지"가 아니라 "**원하는 상태를 Git에 어떻게 적어두지**"로. 배포는 명령이 아니라 선언이 됐다. 그리고 손으로 하던 걸 시스템한테 넘길 때마다, 사람이 실수할 여지가 하나씩 사라졌다. 인프라도 결국 "사람이 까먹고 실수하는 것"을 없애는 게 절반이다.
