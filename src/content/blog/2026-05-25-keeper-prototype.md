기획 업무를 하다 보면 백엔드, 프론트엔드, 앱 팀에 요구사항을 넘기기 전에 이게 말이 되는 기획인지 스스로 검증해야 할 때가 많다. 기존에는 정책서 문서와 Figma 화면으로만 리뷰를 진행했는데, 화면 상태가 복잡해지면 문서만으로는 동작을 완벽하게 시뮬레이션하기 어려웠다.

글로만 써두면 읽는 사람마다 머릿속에 그리는 화면이 달라서, 결국 직접 클릭해 볼 수 있는 프로토타입을 만들어서 정책서와 나란히 두고 검증하기로 했다.

## 기술 스택과 폴더 구조

처음엔 Next.js로 레포를 하나 파서 모든 기획자가 같이 버전 관리를 하는 거창한 그림을 그렸다. 하지만 비개발자들이 git으로 코드를 관리하는 건 무리일 것 같아, 일단 나 혼자 가볍게 쓸 수 있도록 서버가 필요 없는 Vite + React 조합으로 변경했다. 

프로토타입 레포의 핵심은 기능(에픽)별로 철저하게 폴더를 분리하는 것이었다.

```text
src/features/KP-xxxx/
├── index.tsx          # 메인 페이지
├── docs.tsx           # 정책서 전체보기 (TOC 사이드바)
├── components/        # 이 피처 전용 컴포넌트
├── lib/{도메인}/       # types.ts + mock-data.ts
└── policy/spec.md     # 정책서
```

기능 하나당 폴더 하나를 할당하고, 폴더 외부 코드는 절대 건드리지 못하게 규칙을 정했다. 
이렇게 격리해두니 나중에 AI 코딩 어시스턴트에게 수정을 맡길 때도 엉뚱한 곳을 건드리지 않아 관리가 훨씬 편했다. 

## 프로토타입 옆에 정책서 붙이기

검증을 쉽게 하려면 화면과 정책서를 한눈에 보는 게 중요했다. 그래서 화면 오른쪽에 항상 마크다운으로 작성된 정책서(`spec.md`)가 보이도록 구성했다. 

Vite에서 제공하는 `?raw` import를 사용하니 꽤 쉽게 구현할 수 있었다.

```tsx
import policyContent from "./policy/spec.md?raw";
// react-markdown 으로 렌더링
```

이렇게 화면을 클릭해 보면서 바로 옆의 정책서 내용과 동작이 일치하는지 눈으로 대조할 수 있어서 리뷰 효율이 확 올라갔다. 

## 개념 먼저, 화면은 나중에 

프로토타입을 만들면서 가장 크게 배운 점은 "정의가 명확하지 않은 상태에서 화면부터 그리면 안 된다"는 것이었다. 

이전에는 대략적인 구상만 가지고 화면을 먼저 그렸다가, 막상 기획 리뷰를 해보면 현실의 비즈니스 로직과 어긋나서 갈아엎는 경우가 많았다. 
그래서 아예 작업 순서를 강제했다. 브레인스토밍 문서에 '왜 이 기능이 필요한지' 개념을 정리하고, 프로토타입으로 '어떻게 동작할지' 감을 잡은 뒤, 마지막에 정책서에 '무엇을 할 것인지' 적도록 했다. 

## Mock Data의 중요성 

한 번은 에러 경고창 로직을 다 짜뒀는데 화면에 안 나와서 한참 고생한 적이 있다. 알고 보니 로직은 문제없었는데 그 경고창을 띄워줄 만한 엣지 케이스 데이터(Mock Data)가 하나도 없었던 탓이었다. 

결국 프로토타입은 내가 심어둔 Mock Data의 범위 내에서만 동작하기 때문에, 다양한 예외 케이스를 반영한 데이터를 준비하는 게 로직을 짜는 것만큼이나 중요하다는 걸 새삼 느꼈다. 

## 마무리 

요즘은 새로운 기능을 구상할 때 프로토타입 URL을 기획 산출물로 공유하고 있다. 개발자분들과 리뷰할 때도 단순 문서가 아니라 실제로 동작하는 화면을 보며 이야기하니 커뮤니케이션 미스가 눈에 띄게 줄었다. 

물론 매번 프로토타입을 만드는 데 시간이 들긴 하지만, 나중에 개발 단계에서 로직이 꼬여서 수정하는 비용을 생각하면 충분히 가치 있는 작업인 것 같다!
