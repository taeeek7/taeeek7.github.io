운영팀이 쓸 어드민이 계속 필요했다. 키퍼 배정, 스케줄, 세탁 CS, 린넨 재고… 화면이 계속 늘어나는데, 이걸 매번 React로 새로 짜기엔 사람도 시간도 없었다.

그래서 방향을 틀었다. **화면을 코드가 아니라 YAML로 선언하는** 방식(SDUI, Server-Driven UI)으로 갔다.

## 문제부터: "화면이 끝없이 늘어난다"

운영 어드민의 진짜 문제는 화면 하나하나가 어려워서가 아니다. **비슷비슷한 화면이 끝없이 늘어난다**는 거다. 조회하고, 필터 걸고, 몇 건 골라서 상태 바꾸고, CSV로 내리고. 화면마다 하는 일은 거의 똑같은데 테이블과 조건만 다르다.

이걸 React로 짜면 화면 하나에 라우팅, 폼 상태, 테이블 컴포넌트, API 훅, 로딩·에러 처리를 매번 다시 깐다. 화면 열두 개면 이 뻔한 보일러플레이트를 열두 번 반복한다. 게다가 운영 요구사항은 자주 바뀐다. "이 컬럼도 보이게 해주세요", "이 필터 하나만 추가요" 같은 게 매주 온다. 그때마다 프론트 코드 고치고, 빌드하고, 배포하고, 리뷰받는다. 정작 바뀐 건 컬럼 한 줄인데.

데이터 엔지니어인 내가 이 어드민을 떠안았을 때, 프론트 화면 수십 개를 손으로 찍어낼 여력은 애초에 없었다. 그래서 질문을 바꿨다. **"이 화면을 어떻게 잘 짤까"가 아니라 "이 화면을 안 짜고 만들 방법은 없나"** 로.

## 왜 YAML이었나

핵심은 "앱 배포 없이 운영 화면을 바꾸고 싶다"였다. 화면의 공통 구조(조회 폼 → 테이블 → 액션)를 렌더 엔진이 담당하고, 나는 **화면마다 다른 부분만 데이터로 기술**하면 되는 구조. 그게 선언형이고, 그 데이터 포맷으로 YAML을 골랐다.

YAML로 UI 구조·쿼리·API 호출·권한을 다 선언해두면, 설정만 고쳐서 즉시 반영된다. 예를 들면 이런 식이다.

```yaml
- type: query
  resource: mysql.ops
  sqlType: select
  showDownload: csv xlsx
  sql: >
    SELECT id, name, grade, phone
    FROM staff WHERE branch_code = :branch_code
  params:
    - key: branch_code
      valueFromSearch: branch_code
  columns:
    grade: { label: "등급", format: badge }
    phone: { label: "연락처", masking: true }
```

이 몇 줄이면 조회 화면 하나가 뜬다. `:branch_code`는 검색 폼에서 넣은 값이 바인딩되고, CSV·XLSX 다운로드 버튼은 `showDownload` 한 줄로 붙고, 개인정보 마스킹도 옵션 한 줄이다. React였으면 컴포넌트 몇 개와 훅 몇 개가 필요했을 게, 선언 열 줄로 끝난다.

결국 keeper-console 하나가 **24,000줄 YAML**까지 커졌다. 파일 56개, 배정·키퍼·세탁·린넨 도메인을 다 담았다. 같은 걸 React로 짰으면 몇 배는 걸렸을 거다. 개발 기간을 체감상 70~80% 줄였다.

## 렌더 엔진이 YAML을 화면으로 바꾸는 방식

이 플랫폼의 핵심은 "블록(block)" 개념이다. YAML은 결국 **블록의 리스트**고, 블록마다 `type`이 있다. 엔진은 이 타입을 보고 뭘 렌더링할지, 뭘 실행할지 정한다. 내가 실제로 제일 많이 쓴 타입은 셋이었다.

- **`query`** — MySQL에 SELECT/INSERT/UPDATE를 날린다. `sql` 안에서 `:param`으로 파라미터를 바인딩하고, 결과를 `columns` 정의에 따라 테이블로 그린다. 배열은 `IN (:...id_list)` 문법으로 넘긴다.
- **`http`** — 외부 API를 호출한다. 배정 AI를 돌리거나 데이터 파이프라인을 트리거하는 것 같은, DB만으론 안 되는 일을 여기서 처리한다.
- **`submitFn` / `responseFn` / `formatFn`** — 순수 선언으로 안 되는 로직을 끼워넣는 자리다. 버튼을 누르면 뭘 할지(`submitFn`), API 응답을 어떻게 후처리할지(`responseFn`), 셀 값을 어떻게 보여줄지(`formatFn`)를 YAML 안에 JavaScript로 적는다.

파라미터가 어디서 오는지도 선언으로 푼다. `valueFromSearch`(검색 폼), `valueFromRow`(선택한 행), `valueFromEnv`(환경변수), `valueFromUserProperty`(로그인 사용자 속성). 엔진이 이 규칙을 보고 값을 끌어와 쿼리·API에 꽂아준다. 그래서 "이 화면은 로그인한 파트너의 소속 지점만 보여줘" 같은 것도 코드가 아니라 선언 한 줄이 된다.

한 가지 설계적으로 마음에 들었던 건 **권한(RBAC)도 데이터로 선언한다**는 점이다.

```yaml
roles:
  list: [eleven, partners, client]
  view: [eleven, partners]
```

목록을 볼 수 있는 역할과 상세를 볼 수 있는 역할을 이렇게 나눈다. 여기에 쿼리 안에서 `(case when :pcode = '본사' then 1=1 else company_name = :pcode end)` 같은 조건을 걸면, **같은 화면 정의 하나로 본사는 전체를, 파트너는 자기 데이터만** 보게 만들 수 있다. 화면을 역할별로 복제할 필요가 없다. 이 "한 벌의 선언으로 여러 역할을 커버한다"가 선언형의 진짜 힘이었다.

## 제일 효과 봤던 것: 키퍼 스케줄 셀프서비스

이 중에 임팩트가 제일 컸던 건 키퍼 스케줄 페이지였다. 원래는 키퍼 수십 명의 주간 근무 일정을 카카오 채널로 개별 문의받아서 손으로 관리했다. 실수도 잦고 CS도 많았다.

이걸 **키퍼가 직접 웹에서 근무 가능 일자를 등록하는** 셀프서비스로 바꿨다. 키퍼는 앱 계정이 없으니 별도 로그인을 붙이는 건 과했다. 그래서 키퍼별로 토큰을 발급해 **개인 링크만으로 로그인 없이 들어와 등록**하게 했다. 관리자 화면(어드민)에서는 이 스케줄을 조회·확인하고, 등록이 안 된 키퍼에게는 링크를 다시 뿌리는 식으로 운영했다. 관련 CS 문의가 절반으로 줄었다.

재밌는 건, 관리자용 조회 화면은 YAML로 찍어냈지만 **키퍼가 실제로 입력하는 페이지는 따로 만들었다**는 거다. 선언형이 강한 곳(내부 운영자용 CRUD)과 약한 곳(외부 사용자용 입력 UX)의 경계가 여기서 딱 드러났다.

## AI 배정 대시보드는 "결과를 보여주는" 쪽

어드민 안에 AI 배정 대시보드도 붙였다. 여기서 분명히 해두면, **배정 엔진(최적 배정을 계산하는 부분)은 백엔드 팀이 만들었고**, 내가 만든 건 그 결과를 관리자가 확인·조정하는 화면이다. 어드민에서 지점을 고르고 배정을 실행하면, `http` 블록이 데이터 파이프라인 쪽 API를 트리거해 모델을 돌리고, 결과가 DB에 쌓이면 `query` 블록이 그걸 다시 읽어와 화면에 뿌리는 구조다.

배정 결과를 그냥 보여주기만 하면 관리자가 믿기 어렵다. 그래서 패널티 점수(면적·층수·업무량 편차)를 같이 시각화해서, "왜 이렇게 배정됐는지"를 근거로 보여주고 필요하면 수동 조정·확정하게 했다. AI 결과를 사람이 검증할 수 있게 만드는 게 포인트였다. 이 "원본을 보여주고, 사람이 조정본을 만들어 확정한다"는 흐름은 배정 프로토타입에서 배운 걸 그대로 화면으로 옮긴 것이다.

## 삽질과 한계: YAML은 공짜가 아니었다

선언형은 "빠르게 많이" 만들 때 확실히 강하다. 근데 24,000줄까지 키워보니 대가도 분명했다.

**첫째, 로직이 복잡해지면 YAML 안에 JS를 욱여넣게 된다.** `submitFn`, `responseFn` 안에 몇십 줄짜리 자바스크립트가 들어가기 시작하면, 그건 이미 선언이 아니라 코드다. YAML 문자열 블록 안에 든 코드라 IDE의 타입 체크도, 린트도, 자동완성도 없다. 오타 하나가 런타임에 조용히 터진다. 디버깅은 결국 `responseFn`에 `console.log`를 심고 브라우저 콘솔을 들여다보는 원시적인 방식으로 돌아갔다. "코드를 안 짜려고" 선언형을 골랐는데, 정신 차려보면 타입 안전성 없는 환경에서 코드를 짜고 있었다.

**둘째, 선언형은 "화면당 파일"이라 로직 재사용이 약하다.** 같은 상태 전이 처리, 같은 응답 후처리를 여러 파일에 복붙하게 된다. 배정 페이지를 개편하면서 기존 버전을 못 지우고 `-renewal` 붙인 신규 버전을 나란히 두는 일도 생겼다(구버전 1,786줄 + 신규 3,433줄이 한동안 공존했다). 코드였다면 함수로 뺐을 걸, YAML에선 파일이 통째로 불어난다.

**셋째, 24,000줄은 그 자체로 유지보수 대상이다.** 처음엔 "React 안 짜서 좋다"였는데, 나중엔 이 방대한 YAML의 구조를 아는 사람이 나뿐이라는 게 리스크가 됐다. 선언형은 진입 장벽을 낮추는 것처럼 보이지만, 규모가 커지면 결국 "이 플랫폼의 문법과 관례를 아는가"라는 새로운 장벽을 만든다.

## 배운 것

결국 제일 중요했던 건 **"어디까지 선언으로 두고 어디부터 코드로 뺄지"의 경계 감각**이었다. 뻔한 CRUD 조회 화면은 선언형이 압도적으로 낫다. 반대로 복잡한 상태 전이나 외부 사용자용 UX는 선언 안에 억지로 코드를 밀어넣는 것보다 제대로 된 코드로 빼는 게 낫다.

선언형을 도입한다는 건 "코드를 안 짠다"가 아니라 **"코드를 짜야 하는 경계를 스스로 계속 다시 그린다"**는 뜻이었다. 그 경계를 잘 그리면 화면 수십 개를 며칠에 찍어내고, 잘못 그리면 타입 체크도 없는 24,000줄짜리 코드 뭉치를 떠안는다. 이 프로젝트는 그 둘을 다 경험하게 해줬다.
