운영팀이 쓸 어드민이 계속 필요했다. 키퍼 배정, 스케줄, 세탁 CS, 린넨 재고… 화면이 계속 늘어나는데, 이걸 매번 React로 새로 짜기엔 사람도 시간도 없었다.

그래서 방향을 틀었다. **화면을 코드가 아니라 YAML로 선언하는** 방식(SDUI, Server-Driven UI)으로 갔다.

## 왜 YAML이었나

핵심은 "앱 배포 없이 운영 화면을 바꾸고 싶다"였다. 운영은 요구사항이 자주 바뀐다. 컬럼 하나 추가, 필터 하나 변경 같은 걸 매번 프론트 빌드·배포로 처리하면 너무 느리다.

YAML로 UI 구조·쿼리·API 호출·권한을 다 선언해두면, 설정만 고쳐서 즉시 반영된다. 예를 들면 이런 식이다.

```yaml
- type: query
  resource: mysql.ops
  sql: >
    SELECT id, name, grade, phone
    FROM staff WHERE branch_code = :branch_code
  columns:
    grade: { label: "등급", format: badge }
    phone: { label: "연락처", masking: true }
```

이 몇 줄이면 조회 화면 하나가 뜬다. 개인정보 마스킹 같은 것도 옵션 한 줄이다. 결국 keeper-console 하나가 **24,000줄 YAML**까지 커졌는데, 같은 걸 React로 짰으면 몇 배는 걸렸을 거다. 개발 기간을 체감상 70~80% 줄였다.

## 제일 효과 봤던 것: 키퍼 스케줄 셀프서비스

이 중에 임팩트가 제일 컸던 건 키퍼 스케줄 페이지였다. 원래는 키퍼 수십 명의 주간 근무 일정을 카카오 채널로 개별 문의받아서 손으로 관리했다. 실수도 잦고 CS도 많았다.

이걸 **키퍼가 직접 웹에서 근무 가능 일자를 등록하는** 셀프서비스로 바꿨다. 별도 로그인도 없이 개인 링크(Fernet 토큰)만으로 들어와서 등록하게 했다. 관련 CS 문의가 절반으로 줄었다.

## AI 배정 대시보드는 "결과를 보여주는" 쪽

어드민 안에 AI 배정 대시보드도 붙였다. 여기서 분명히 해두면, **배정 엔진(LLM으로 최적 배정을 계산하는 부분)은 백엔드 팀이 만들었고**, 내가 만든 건 그 결과를 관리자가 확인·조정하는 화면이다.

배정 결과를 그냥 보여주기만 하면 관리자가 믿기 어렵다. 그래서 패널티 점수(면적·층수·업무량 편차)를 같이 시각화해서, "왜 이렇게 배정됐는지"를 근거로 보여주고 필요하면 수동 조정·확정하게 했다. AI 결과를 사람이 검증할 수 있게 만드는 게 포인트였다.

## 배운 것

YAML 선언형은 "빠르게 많이" 만들 때 확실히 강하다. 다만 로직이 복잡해지면 YAML 안에 JS를 욱여넣게 되는데(submitFn, responseFn), 이게 길어지면 오히려 관리가 어려워진다. "어디까지 선언으로 두고 어디부터 코드로 뺄지"의 경계 감각이 결국 제일 중요했다.
