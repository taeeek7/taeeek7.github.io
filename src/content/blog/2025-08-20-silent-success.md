데이터 파이프라인 짜던 사람이 갑자기 UI 문구를 뜯어고치게 됐다. 처음엔 "이게 그렇게 중요한가?" 싶었는데, 막상 우리 서비스 화면을 하나하나 뜯어보니까 생각이 바뀌었다.

제일 먼저 눈에 띈 게 **성공 알림**이었다.

## "저장되었습니다"를 꼭 띄워야 하나?

버튼 하나 누를 때마다 하단에 스낵바가 뜬다.

- "워크스페이스가 생성되었습니다."
- "키퍼가 배정되었습니다."
- "수정 내용이 저장되었습니다."

처음엔 당연한 거라고 생각했다. 근데 곰곰이 보니, **이 알림들이 알려주는 정보가 이미 화면에 다 나와 있었다.** 저장을 누르면 모달이 닫히고, 버튼이 비활성화되고, 데이터가 갱신된다. 사용자는 이미 "됐구나"를 안다. 그 위에 굳이 스낵바를 하나 더 띄우는 건, 사용자 시선을 아래로 한 번 끌어내리는 방해였다.

그래서 원칙을 하나 세웠다.

> **성공은 조용히. 실패만 시끄럽게.** (Silent Success)

성공은 UI 상태 변화로 충분히 전달된다. 굳이 말로 안 해도 된다. 이 원칙으로 성공 알림 **158개를 전부 지웠다.** 지우고 나니 화면이 훨씬 조용해졌다.

## 에러 메시지는 짧게

반대로 실패는 알려줘야 한다. 근데 이것도 말이 너무 길었다.

| Before | After |
|--------|-------|
| 우선 순위를 긴급으로 변경하는데 실패했습니다. | 우선 수행 요청 실패 |
| 수정 내용 저장에 실패했습니다. 다시 시도해 주세요. | 저장 실패 |

"다시 시도해 주세요"는 안 적어도 다들 다시 누른다. 자명한 안내는 뺐다. 마침표도 뺐다. 명사형으로 끝냈다. 짧을수록 빨리 읽힌다.

## 진짜 어려웠던 건 용어였다

문구 다듬는 것보다 훨씬 골치였던 게 **용어**다. 같은 걸 화면마다 다르게 부르고 있었다.

- 어디선 "업무", 어디선 "일감"
- 어디선 "할일", 어디선 "체크리스트"
- 어디선 "공간 그룹", 어디선 "지점"

사용자 입장에서 이건 매번 새로 배워야 하는 비용이다. "업무랑 일감이 같은 건가?" 하는 순간 흐름이 끊긴다. 그래서 **104개 용어를 하나로 통일**했다. 업무 → 일감, 공간 그룹 → 지점, 할일 → 체크리스트.

이게 왜 데이터 하던 나한테 의외로 익숙했냐면, **결국 스키마 정규화랑 똑같은 문제**였기 때문이다. 같은 개념은 같은 이름으로. 이름이 흔들리면 데이터도 사람도 헷갈린다.

## 남은 것

문구를 다 고치고 나서는 이걸 어떻게 유지·번역할지가 문제였다. 그래서 Lokalise를 붙이고 `메뉴.영역.대상.타입` 같은 계층 키 체계를 세워서, 10,000개 넘는 문자열을 기획자·디자이너도 직접 관리할 수 있게 만들었다. 이 얘긴 다음에 따로 정리해야겠다.

정리하면, UX 라이팅은 "예쁜 말 고르기"가 아니었다. **뭘 안 보여줄지 정하는 일**에 가까웠다. 158개를 지운 게 제일 큰 개선이었다는 게 좀 아이러니하다.
