데이터 파이프라인 짜던 사람이 갑자기 UI 문구를 뜯어고치게 됐다. 처음엔 "이게 그렇게 중요한가?" 싶었는데, 막상 우리 서비스 화면을 하나하나 뜯어보니까 생각이 바뀌었다.

바뀐 이유는 이거였다. 나는 계속 "어떤 문구가 더 예쁜가"를 묻고 있었는데, 화면을 열어놓고 보니 진짜 문제는 문구 하나하나의 완성도가 아니었다. **화면 전체가 너무 시끄러웠고, 같은 걸 화면마다 다른 이름으로 부르고 있었다.** 예쁜 말을 고르는 문제가 아니라, 사용자가 이 화면을 읽는 데 드는 비용의 문제였다. 그래서 처음부터 질문을 바꿔서 시작했다. "무슨 말을 할까"가 아니라 **"사용자가 뭘 안 읽어도 되게 만들까"**.

제일 먼저 눈에 띈 게 **성공 알림**이었다.

## "저장되었습니다"를 꼭 띄워야 하나?

버튼 하나 누를 때마다 하단에 스낵바가 뜬다.

- "워크스페이스가 생성되었습니다."
- "키퍼가 배정되었습니다."
- "수정 내용이 저장되었습니다."

처음엔 당연한 거라고 생각했다. 근데 곰곰이 보니, **이 알림들이 알려주는 정보가 이미 화면에 다 나와 있었다.** 저장을 누르면 모달이 닫히고, 버튼이 비활성화되고, 데이터가 갱신된다. 사용자는 이미 "됐구나"를 안다. 그 위에 굳이 스낵바를 하나 더 띄우는 건, 사용자 시선을 아래로 한 번 끌어내리는 방해였다.

키퍼 어드민은 운영자가 하루에 배정하고, 수정하고, 저장하는 걸 수백 번 반복하는 도구다. 한 번의 스낵바는 아무것도 아니지만, 그게 하루에 수백 번이면 얘기가 다르다. **성공 알림은 "잘됐다"는 안심을 주는 대신, 매번 "이거 읽어야 하나?"라는 아주 작은 판단을 사용자한테 시킨다.** 그 판단이 쌓이는 게 인지 부하다. 정보가 없어서 생기는 부하가 아니라, 안 봐도 되는 정보를 걸러내느라 생기는 부하다.

그래서 원칙을 하나 세웠다.

> **성공은 조용히. 실패만 시끄럽게.** (Silent Success)

기준은 단순하다. **알림이 전달하려는 정보가 이미 UI 상태 변화로 드러나 있으면, 그 알림은 중복이다.** 모달이 닫혔으면 저장은 된 거다. 목록에 새 항목이 보이면 추가는 된 거다. 성공은 화면이 이미 말하고 있으니 말로 또 할 필요가 없다. 반대로 실패는 UI만 봐선 알 수 없다 — 뭐가 잘못됐는지는 상태 변화로 안 드러나니까, 실패만 시끄럽게 남겼다.

이 원칙으로 성공 알림 **158개를 전부 지웠다.** 지우고 나니 화면이 훨씬 조용해졌다. 재밌는 건, 지운 뒤에 "성공했는지 모르겠다"는 불편이 거의 안 나왔다는 거다. 애초에 그 알림들이 정보를 주고 있던 게 아니었다는 방증이다.

물론 예외는 남겼다. 결과가 화면에 즉시 안 드러나는 비동기 작업 — 예를 들어 백그라운드에서 도는 일괄 처리 같은 건, 성공을 조용히 넘기면 사용자가 정말 몰라서 다시 누른다. "성공은 조용히"는 규칙이 아니라 **"UI가 이미 말하고 있는가"를 매번 되묻는 기준**이었다. 기계적으로 다 지운 게 아니라, 하나씩 "이건 화면이 말해주나?"를 물어보고 지웠다.

## 에러 메시지는 짧게

반대로 실패는 알려줘야 한다. 근데 이것도 말이 너무 길었다.

| Before | After |
|--------|-------|
| 우선 순위를 긴급으로 변경하는데 실패했습니다. | 우선 수행 요청 실패 |
| 수정 내용 저장에 실패했습니다. 다시 시도해 주세요. | 저장 실패 |

"다시 시도해 주세요"는 안 적어도 다들 다시 누른다. 자명한 안내는 뺐다. 마침표도 뺐다. 명사형으로 끝냈다. 짧을수록 빨리 읽힌다.

여기도 원칙은 같았다. **사용자가 이미 아는 걸 문장으로 반복하지 않는다.** 에러 메시지의 목적은 "실패했다"를 알리는 것 하나다. 위로도, 사과도, 뻔한 재시도 안내도 그 목적엔 군더더기였다. 50개 넘는 에러 메시지를 이 기준으로 다시 썼다. 유효성 메시지도 같이 손봤다 — "초성, 중성, 종성은 단독으로 사용할 수 없습니다"처럼 개발자 머릿속 용어가 그대로 노출된 것들을 "자음, 모음만 입력할 수 없습니다"로 바꿨다. 사용자는 초성이 뭔지 알 필요가 없다.

## 진짜 어려웠던 건 용어였다

문구 다듬는 것보다 훨씬 골치였던 게 **용어**다. 같은 걸 화면마다 다르게 부르고 있었다.

- 어디선 "업무", 어디선 "일감"
- 어디선 "할일", 어디선 "체크리스트"
- 어디선 "공간 그룹", 어디선 "지점"

사용자 입장에서 이건 매번 새로 배워야 하는 비용이다. "업무랑 일감이 같은 건가?" 하는 순간 흐름이 끊긴다. 그래서 **104개 용어를 하나로 통일**했다. 업무 → 일감, 공간 그룹 → 지점, 할일 → 체크리스트.

이게 왜 데이터 하던 나한테 의외로 익숙했냐면, **결국 스키마 정규화랑 똑같은 문제**였기 때문이다. 같은 개념은 같은 이름으로. 이름이 흔들리면 데이터도 사람도 헷갈린다. DB에서 같은 엔티티를 테이블마다 다른 컬럼명으로 부르면 조인이 지옥이 되는 것처럼, UI에서 같은 개념을 화면마다 다르게 부르면 사용자 머릿속에서 개념이 조인이 안 된다.

그리고 이건 문구 고치기보다 훨씬 무거운 일이었다. 문구는 그 자리에서 고치면 끝인데, **용어는 하나 바꾸면 그 단어가 박힌 모든 화면·모든 코드가 같이 움직여야** 한다. "업무"를 "일감"으로 바꾸기로 정하는 순간, 그게 등장하는 수십 개 화면과 그 밑의 문자열을 전부 같이 갈아야 했다. 용어 통일을 진짜로 지속 가능하게 만들려면, 문자열을 코드에서 떼어내서 **한 군데서 관리**할 수 있어야 했다. 여기서 다음 문제가 시작됐다.

## 문자열을 코드에서 떼어내기

문구와 용어를 다 정리하고 나니 남은 질문은 이거였다. **이걸 어떻게 유지하고, 어떻게 번역하지?**

지금까지는 문자열이 코드 곳곳에 하드코딩돼 있었다. 이 상태에선 세 가지가 다 안 된다. (1) 용어 하나 바꾸려면 개발자가 코드를 뒤져야 하고, (2) 기획자·디자이너는 자기가 정한 문구가 실제로 뭐로 들어갔는지 볼 수가 없고, (3) 다국어는 시작도 못 한다. 그래서 문자열을 전부 코드 밖으로 빼서 **키로 참조**하게 만들기로 했다. 코드는 `t("공통.버튼.저장.텍스트")`만 부르고, 실제 문자열은 별도 저장소에서 관리한다. 저장소로는 Lokalise를 붙였다.

문제는 문자열이 어드민 8,600여 개, 앱 1,900여 개, 합쳐서 **10,000개가 넘었다**는 거다. 이 규모에서 제일 먼저 정해야 할 건 번역 도구가 아니라 **키를 뭐라고 부를지**였다. 키 체계가 곧 이 프로젝트의 스키마였다.

## 키 이름을 한글로 지은 이유

키 네이밍에서 내가 내린 제일 큰 결정은, **키를 한글로 짓는다**는 거였다.

```
{도메인}.{UI컨텍스트}.{설명}.{타입}

공간관리.모달.공간수정.제목
완료취소내역.필터.조회기준선택.항목
공통.유효성.문자열.최대길이
```

4단계 점 표기법이다. 앞에서부터 **어느 화면인지(도메인) → 화면 안 어디인지(UI 컨텍스트) → 무슨 내용인지(설명) → 어떤 종류인지(타입)** 순으로 좁혀 들어간다. 여러 화면이 공유하는 문자열은 도메인을 `공통`으로 두고 한 군데로 모았다 — `공통.버튼.닫기`, `공통.상태.미배정`처럼.

영어로 지을 수도 있었다. `space.modal.edit.title` 같은 식으로. 근데 그렇게 안 한 이유가 이 프로젝트의 핵심이었다. **이 키를 실제로 관리할 사람이 개발자가 아니라 기획자와 디자이너였기 때문이다.** 10,000개 문자열을 개발자가 병목이 돼서 관리하면 그 체계는 오래 못 간다. 기획자가 문구를 정하고, 그 자리에서 바로 키를 읽고 어디 쓰이는 문자열인지 알 수 있어야 했다. `space.modal.edit.title`은 비개발자한테 한 번 번역해서 읽어야 하는 텍스트지만, `공간관리.모달.공간수정.제목`은 그냥 읽힌다.

키 이름은 "기계가 읽기 좋은 것"과 "사람이 읽기 좋은 것" 사이의 선택인데, 여기선 후자를 골랐다. 이 문자열들을 실제로 매일 만질 사람이 사람이었으니까. 타입도 한글로 통일했다 — `제목`, `라벨`, `텍스트`, `안내`, `에러`, `유효성`, `플레이스홀더`, `항목`. 스낵바로 나가는 에러는 `.에러`, 폼 검증 메시지는 `.유효성`으로 구분해서, 키만 봐도 이게 화면 어디에 어떤 모양으로 뜨는 문자열인지 짐작되게 했다.

새 문자열은 주로 Figma에서 나왔다. 디자인 시안에서 텍스트를 뽑아, 이 4단계 규칙으로 키를 붙여 CSV로 정리한 뒤 파이프라인에 태우는 흐름이었다. 화면(도메인)과 컴포넌트(UI 컨텍스트)가 시안에 이미 다 있으니, 키의 앞 두 단계는 시안을 보면 거의 정해졌다. 규칙을 화면 구조에 맞춰 짠 덕분이다.

## 10,000개를 사람 손 안 타게

키 체계를 정하고 나서는 이걸 **자동화**했다. 안 그러면 문자열 하나 바뀔 때마다 사람이 복사·붙여넣기를 하다가 반드시 사고가 난다.

원본은 구글 스프레드시트 한 장이다. 기획자·디자이너가 여기서 키와 한국어 문구, 그리고 메모·태그를 관리한다. 여기서부터 코드까지는 스크립트가 잇는다.

```
구글 스프레드시트 (기획/디자인이 관리)
    ↓ Python 변환 스크립트 (gspread)
JSON / .properties  (Lokalise 업로드 포맷)
    ↓ Lokalise CLI upload
Lokalise (웹에서 번역 관리)
    ↓ Lokalise CLI download
JSON  (앱·어드민 코드에 적용)
```

변환 스크립트는 시트를 읽어서 플랫폼별 포맷으로 뱉는다. 어드민·앱은 JSON, 서버는 `.properties`. 같은 시트에서 나오지만 소비하는 쪽 포맷이 다르니 타겟별로 갈랐다. 여기에 `--cleanup-mode`를 걸어서, 시트에서 지운 키는 Lokalise에서도 지워지게 했다 — 코드에 유령 키가 쌓이는 걸 막으려고. 그리고 업로드마다 날짜 태그(예: `20260212`)를 붙였다. 이러면 "이번 릴리즈에 어떤 문자열이 바뀌었나"를 태그로 되짚을 수 있다. 문자열도 결국 버전 관리가 필요한 자산이라, 배포 단위로 꼬리표를 달아둔 거다.

## 삽질: 업로드는 됐다는데, 뭐가 됐는지를 모른다

여기서 은근히 애먹은 게 있다. Lokalise CLI로 파일을 올리면 응답에 `process_id` 하나만 딸랑 온다. 업로드가 **비동기**라, "몇 개가 새로 들어갔고 몇 개가 바뀌었는지"는 그 자리에서 안 알려준다. 10,000개를 올려놓고 실제로 뭐가 반영됐는지를 모르면, 이 자동화는 믿고 쓸 수가 없다.

CLI에 결과를 조회하는 명령이 있길래 써봤는데, 정작 상세 내역(`details`)이 **빈 객체로 왔다.** 여기서 한참 헤맸다. 결국 CLI를 버리고 REST API를 직접 폴링했다 — process 상태가 `finished`가 될 때까지 2초 간격으로 두들기고, 완료되면 `key_count_inserted / updated / skipped`를 뽑아서 출력하게 했다.

```
Inserted: 12 | Updated: 43 | Skipped: 1191
```

이 한 줄을 보려고 API를 직접 붙인 거다. 근데 이 한 줄이 있고 없고가 컸다. **자동화에서 제일 무서운 건 실패가 아니라, 됐는지 안 됐는지 모르는 상태다.** 실패는 고치면 되는데, 조용히 안 반영된 건 나중에 사용자 화면에서 터진다. 그래서 굳이 삽질을 해서라도 "몇 개가 어떻게 됐다"를 눈으로 확인하는 관문을 만들어뒀다. 어디서 많이 본 패턴이다 싶었는데, 앞선 배정 프로토타입에서 LLM 출력을 곧이곧대로 안 믿고 **파싱 → 검증 → 저장** 관문을 세웠던 거랑 똑같았다. 자동화든 AI든, 결과를 확인하는 관문 없이 믿으면 반드시 뒤통수를 맞는다.

토큰 얘기도 하나. Lokalise API 토큰은 스크립트나 `.env`에 박지 않고 macOS 키체인에서 꺼내 쓰게 했다. `security find-generic-password`로 실행할 때만 읽는다. 공개 저장소에 토큰 커밋하는 사고를 원천 봉쇄하려는 거였고, 이건 데이터 하면서 시크릿 관리로 여러 번 데어본 습관이 그대로 나온 거다.

## 남은 것, 그리고 배운 것

자동화 파이프라인까지 세웠지만 한계도 분명하다. 원본이 스프레드시트 한 장이라, 이게 곧 단일 실패 지점(SPOF)이다. 누가 실수로 키 하나를 지우거나 오타를 내면 그대로 흘러간다 — `--cleanup-mode` 때문에 지운 키는 진짜로 지워지니까 더 그렇다. 시트 단계에서 키 형식이나 중복을 검증하는 장치는 아직 없다. 다국어도 지금은 한국어 기준으로 틀만 깔아둔 거지, 실제 번역 품질을 관리하는 프로세스는 다음 숙제다.

정리하면, UX 라이팅은 "예쁜 말 고르기"가 아니었다. **뭘 안 보여줄지 정하는 일**에 가까웠고, 그 뒤엔 **같은 개념을 같은 이름으로 부르게 만들고, 그걸 사람 손 안 타게 유지하는 시스템 설계**가 있었다. 158개 성공 알림을 지운 게, 250개 넘는 문자열을 새로 쓴 것보다 큰 개선이었다는 게 좀 아이러니하다. 결국 좋은 문구를 더한 게 아니라, 안 좋은 소음을 뺀 쪽이 사용자한테 더 나은 화면을 만들었다.
