데일리 스크럼을 Geekbot 같은 유료 SaaS로 돌리고 있었는데, 두 가지가 걸렸다. 하나는 월 구독료, 하나는 "우리 팀 방식대로 못 바꾸는" 답답함. 그래서 직접 만들었다. 이름은 Scrumble.

## 왜 SaaS를 버렸나

구독료는 사실 방아쇠였지, 진짜 이유는 아니었다. 팀 인원이 늘면 요금제가 한 칸씩 올라가는 게 눈에 밟히긴 했지만, 그거 아끼자고 며칠씩 봇을 짜는 건 계산이 안 맞는다. 진짜로 걸린 건 세 가지였다.

- **커스터마이즈가 막힌다.** 우리는 한국 공휴일엔 안 보내고 싶고, 대체공휴일도 챙기고 싶고, "오늘 휴가"를 버튼 하나로 처리하고 싶었다. SaaS의 설정 화면은 딱 그들이 정해둔 만큼만 열려 있다.
- **데이터가 남의 서버에 있다.** 스탠드업은 은근히 팀의 로그다. 누가 언제 뭘 했는지, 무슨 블로커가 있었는지가 쌓인다. 이걸 나중에 들여다보려면 내 DB에 있어야 한다.
- **질문 흐름을 내가 못 짠다.** 이게 제일 컸다. 스탠드업 도구는 대부분 폼이다. 칸 채우는 느낌이라 딱딱하고, 솔직히 참여율이 잘 안 나온다.

그래서 문제를 "구독 끊기"가 아니라 **"우리 팀 스탠드업을 처음부터 다시 정의하기"**로 다시 잡았다. 스택은 익숙한 걸로 갔다 — FastAPI + Slack Bolt + APScheduler, DB는 MySQL.

## 폼이 아니라 대화로

기존 도구가 폼이라면, 나는 **DM으로 대화하듯** 묻기로 했다.

```
봇: 오늘 컨디션은 어떤가요?
나: 좋아요!

봇: 오늘의 근무 형태는 어떻게 되시나요?
나: 오전 재택, 오후 출근

봇: 어제 수행한 업무는 어떻게 되었나요?
    📌 어제 계획하신 내용: "API 개발 마무리"
나: 마무리하고 테스트까지 작성했습니다
```

컨디션 → 근무형태 → 어제 한 일 → 오늘 할 일 → 공유사항, 5단계로 하나씩 물어본다. 특히 **"어제 계획"을 리마인더로 같이 보여주는 것**에 신경 썼다. 어제 뭐 하겠다고 했는지 옆에 띄워주니까, 오늘 회고가 자연스럽게 이어진다. 이게 은근히 좋았다. (전날 응답이 없으면 최근 7일 안에서 마지막 "오늘 할 일"을 끌어온다.)

대화형이라 상태를 어딘가 들고 있어야 한다. 나는 **user_id를 키로 하는 인메모리 딕셔너리**에 `step`(0~4)과 지금까지 받은 답을 쌓아뒀다. DM이 하나 들어오면 그 유저의 현재 step에 해당하는 필드에 답을 저장하고 step을 +1 한 뒤, 다음 질문을 던지거나 마지막이면 채널에 게시한다. 즉 봇 자체는 상태가 없고, **누가 지금 몇 번째 질문에 서 있는지**만 기억하는 얇은 상태 기계다. 인메모리라 프로세스가 죽으면 진행 중이던 대화가 날아간다 — 프로덕션이라면 Redis로 빼야 하는 지점인데, 아침 한 번 도는 봇이라 일단 이대로 뒀다. 솔직한 부채다.

## 봇을 두 개의 심장으로 나눴다

만들면서 제일 고민한 게 구조였다. 이 봇은 성격이 다른 두 일을 동시에 해야 한다.

1. **정해진 시간에 먼저 말 걸기** — 아침에 팀원들에게 DM 발송 (능동, 시간 기반)
2. **아무 때나 오는 답 받기** — 유저가 답하고 버튼 누르는 걸 실시간 처리 (수동, 이벤트 기반)

그래서 봇 안에 심장을 둘로 뒀다. 하나는 **APScheduler**, 하나는 **Slack Bolt의 Socket Mode**.

Socket Mode를 고른 이유가 있다. Slack 이벤트를 받는 정석은 공개 HTTP 엔드포인트를 열고 Slack이 그리로 콜백을 쏘게 하는 건데, 그러려면 밖에서 들어오는 인바운드 URL을 뚫어야 한다. Socket Mode는 반대로 **봇이 Slack 쪽으로 WebSocket을 걸어 나가서** 그 위로 이벤트를 받는다. 인바운드 웹훅도, 공개 URL도 필요 없다. 내부에서 돌리는 사이드 봇한테는 이게 훨씬 편했다.

문제는 Socket Mode 핸들러가 `handler.start()`에서 블로킹으로 WebSocket 루프를 물고 있다는 거다. 이걸 FastAPI 메인 흐름에서 그냥 부르면 서버가 거기서 멈춘다. 그래서 **데몬 스레드에 얹어 백그라운드로** 띄웠다. FastAPI는 뜰 때 스케줄러 세팅 → 스케줄러 시작 → Socket Mode 스레드 시작을 순서대로 하고, 내려갈 때 스케줄러를 정리한다. HTTP API, 스케줄러, WebSocket 세 개가 한 프로세스 안에서 각자 자기 루프를 도는 구조다.

## 스케줄러는 크론 한 줄이 아니었다

처음엔 "아침에 함수 한 번 부르면 되지" 싶었는데, 막상 짜보니 신경 쓸 게 꽤 있었다.

- **평일만.** `CronTrigger`에 `day_of_week="mon-fri"`를 걸어 주말은 아예 안 돈다.
- **타임존을 명시.** 스케줄러 자체를 `Asia/Seoul`로 고정했다. 이거 안 하면 서버 타임존에 휘둘린다 (이것 때문에 뒤에서 한 번 크게 뎄다).
- **misfire grace time.** 배포하느라 그 시각에 프로세스가 잠깐 떠 있었으면 발송을 놓친다. 30분 grace를 줘서, 조금 늦게라도 떴으면 그날 스탠드업은 나가게 했다.
- **시각·타임존·채널을 설정으로.** 코드에 박지 않고 설정에서 읽게 했다. 그래서 **테스트 환경에서는 스케줄러를 아예 끄는** 스위치를 둘 수 있었다 — 안 그러면 CI나 로컬에서 봇이 진짜로 팀 채널에 글을 쏜다.

크론과 별개로 **수동 트리거 API**도 하나 팠다. `user_ids`와 `skip_holiday_check` 파라미터를 넘기면, 특정 사람한테만 다시 보내거나 공휴일 체크를 건너뛰고 강제로 발송할 수 있다. "아까 봇이 너만 빼먹었네, 다시 쏴줄게" 같은 상황이 실제로 생기더라.

## 우리 환경에 맞춘 디테일

직접 만드니까 SaaS로는 못 하던 세밀한 걸 넣을 수 있었다.

- **한국 공휴일 자동 스킵**: 주말이랑 법정 공휴일(대체공휴일 포함)엔 안 보낸다. `holidays` 라이브러리를 `observed=True`로 써서 대체공휴일까지 잡고, 2026년 새로 생긴 제헌절은 라이브러리가 아직 몰라서 수동으로 얹어줬다.
- **휴가 버튼**: "오늘 휴가입니다" 누르면 스탠드업 건너뛰고 채널에 상태 게시. 안 그러면 팀원이 "쟤 왜 응답 안 하지?" 한다. 휴가를 취소하면 그날 스탠드업을 다시 시작할 수도 있다.
- **수정하기**: 한번 제출하고 오타 발견하면 곤란하니까, 채널 게시 후에도 모달로 편집 가능하게. 이때 채널에 새 메시지를 달지 않고, 원래 메시지의 타임스탬프를 기억해뒀다가 **그 메시지를 업데이트**해서 중복을 막았다.

## 삽질 포인트

만들면서 예상 못 한 데서 터진 것들. 이게 제일 재밌었다.

**시간이 이모지로 변신해서 사라졌다.** 누가 "10:00에 회의"라고 쓰면 채널엔 "10"만 남고 뒤가 증발했다. 한참 헤맸는데, 범인은 Slack이었다. `:00`, `:30` 같은 걸 이모지 코드로 해석해서 삼켜버린 거다. 해결은 **콜론 사이에 zero-width space를 끼워 넣는 것**. 눈엔 안 보이지만 Slack이 "이건 이모지 이름이 아니네" 하고 넘어간다. 단 `:100:` 같은 진짜 이모지는 살려야 하니까, 앞뒤에 콜론이 없을 때만 바꾸도록 lookbehind/lookahead를 붙였다.

```python
# 10:00 → 10:<zero-width space>00
# 단 :100: 같은 실제 이모지는 앞뒤 콜론 체크로 보호
text = re.sub(r'(?<!:)(\d{1,2}):(\d{2})(?!:)', '\\1:​\\2', text)
```

**날짜가 하루씩 어긋났다.** 응답이 엉뚱한 날짜로 저장되는 걸 발견했다. 코드에서 `date.today()`를 썼는데, 이 함수는 **서버의 시스템 타임존을 따른다.** 배포된 컨테이너는 UTC라서, 한국 아침에 받은 응답이 UTC 기준 아직 전날로 찍혔다. `ZoneInfo("Asia/Seoul")`로 오늘을 계산하는 헬퍼를 만들어서 날짜 쓰는 곳을 전부 갈아끼웠다. 로컬에선 멀쩡하고 배포하면 틀어지는, 딱 짜증나는 종류의 버그였다.

**라이브러리를 올렸더니 import가 깨졌다.** slack-bolt를 1.18에서 1.22로 올렸더니 봇이 뜨자마자 죽었다. `SocketModeHandler`가 `slack_bolt.adapter.socket_mode`에서 `slack_bolt.adapter.socket_mode.builtin`으로 자리를 옮긴 거였다. 고치는 건 한 줄인데, 원인 찾는 게 늘 오래 걸린다. 버전 올릴 땐 changelog를 봐야 한다는 걸 또 배웠다.

**입력 새니타이징.** 사용자가 붙여넣은 텍스트에 NULL 바이트나 제어 문자가 섞여 있으면 DB에 넣다가 터졌다. NULL 바이트·제어 문자를 걷어내고 필드마다 길이를 잘랐다 (근무 형태는 짧은 컬럼, 나머지는 긴 컬럼이라 한도가 다르다). 위의 이모지 이스케이프도 결국 이 새니타이징 함수 한 곳에 다 모아뒀다.

## 아직 못 한 것

솔직히 프로덕션급이라고 부르긴 이르다.

- 대화 상태가 인메모리라 배포·재시작하면 진행 중이던 스탠드업이 날아간다.
- 그래서 인스턴스를 여러 개로 못 늘린다. 지금은 한 대로 충분하지만 구조적 한계다.
- Socket Mode WebSocket이 끊기면 봇이 조용히 죽는다. 재연결·알림을 제대로 붙여야 마음이 놓인다.

이런 건 SaaS 쓸 땐 몰랐던, 직접 만들어야 보이는 것들이다.

## 결과

Geekbot 구독료를 0원으로 만들었고, 무엇보다 우리 팀 프로세스에 맞게 계속 고칠 수 있게 됐다. 공휴일 로직이랑 날짜 계산 같은 핵심엔 테스트도 붙여놔서 마음 편히 굴린다.

돌아보면 "구독료 아끼기"로 시작했는데, 만들면서 "우리 팀한테 좋은 스탠드업이 뭐지?"를 다시 생각하게 된 게 더 큰 소득이었다. 도구를 직접 만들면 그 도구가 풀려는 문제를 다시 보게 된다.
